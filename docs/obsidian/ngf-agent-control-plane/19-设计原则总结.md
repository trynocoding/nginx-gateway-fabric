# 设计原则总结

NGF 与 Agent 的交互不是简单的“控制面调接口让 NGINX reload”。它体现了一组清晰的分层和一致性设计。

## 控制面和数据面分离

NGF 控制面：

- 理解 Kubernetes 和 Gateway API。
- 计算期望状态。
- 生成配置。
- 维护数据面连接。

Agent 数据面：

- 不 watch Gateway API。
- 不参与全局调度。
- 只执行本 Pod 内的配置应用和状态回传。

这种分离降低了数据面权限，也让控制面可以独立演进。

## 期望状态驱动

NGF 不是把每个 watch event 当成一次命令式变更，而是不断计算当前期望状态：

```text
Kubernetes resources
  -> graph
  -> generated config
  -> data plane convergence
```

这符合 Kubernetes controller 模式，也让重试和恢复更自然。

## 长连接承载控制面命令

Agent 主动连接 NGF：

- 避免控制面主动进入 Pod。
- 适合 Pod 动态创建和重启。
- Subscribe 长流支持持续下发。
- 断开后 Agent 可自动重连。

## 文件摘要和按需拉取分离

`Subscribe` 发送命令和文件摘要，`FileService` 提供文件内容。

好处：

- 控制命令和大文件内容分离。
- stream 消息更轻。
- 文件访问可以独立做路径和权限控制。
- 后续扩展文件元数据更清晰。

## 运行态 DeploymentStore

NGF 不只依赖 Kubernetes 对象，还维护数据面运行态：

- 当前配置文件。
- 当前 config version。
- 每个 Pod 的配置状态。
- broadcaster subscribers。

这层运行态是连接 Kubernetes 期望状态与 gRPC 长连接的桥。

## Broadcaster 保证多 Pod 更新

一个 Gateway 数据面 Deployment 可能有多个 Pod。Broadcaster 的设计让同一份配置可以发给所有订阅中的 Agent，并通过 ResponseCh 协调完成状态。

新订阅者通过初始配置拿到当前最新状态，避免必须等待下一次资源变化。

## 锁保护一致性窗口

`Subscribe` 中 `Deployment.FileLock` 的设计很关键。它保护：

- 订阅注册。
- 初始配置读取。
- 并发配置广播。

没有这个锁，新连接 Agent 可能拿到旧配置并错过新更新。

## Agent 插件化

Agent 内部用 MessagePipe 和插件隔离职责：

- CommandPlugin 负责 gRPC。
- Config apply 插件负责配置应用。
- Collector 相关插件负责观测数据。

这避免控制面协议层和本地执行层互相缠绕。

## 安全默认开启

连接安全依赖：

- TLS。
- server_name。
- projected service account token。
- token audience。
- Pod/owner 识别。

这让 NGF 能判断连接来源，而不是只相信网络可达。

## 二开时的判断问题

做任何改动前先问：

1. 这是 Gateway API 语义变化，还是数据面执行变化？
2. 是否需要改 proto？
3. 是否需要改 Agent 配置？
4. 是否影响初始配置和后续广播的一致性？
5. 错误如何回传到 Gateway/Route status？
6. 多 Pod、多连接、重连时行为是否正确？

如果这些问题回答不清楚，先不要动核心链路。
