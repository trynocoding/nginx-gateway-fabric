# 24 接口的隐式实现

> [!abstract] 核心结论
> interface 描述调用方需要的方法集。具体类型无需写 `implements`；其方法集匹配就实现。接口值可理解为“动态类型 + 动态值”，二者都为空才等于 nil，因此装有 typed nil pointer 的接口并不为 nil。

## 学习目标与前置

前置：[[22-值与指针接收者方法集]]、[[23-嵌入和组合]]。完成后应能：

- 写出接口的全部方法签名并判断具体类型是否满足；
- 使用 `var _ I = (*T)(nil)` 做编译期实现断言；
- 解释 nil interface 与 typed nil 的差别；
- 理解“小接口、由 consumer 定义”的方向，并在 [[25-小接口与依赖注入]] 中实践。

## 1. 接口语法与方法集

```go
type Store interface {
	Get(string) (string, bool)
	Put(string, string) error
}
```

这两个方法的名称、参数、结果必须完全匹配；参数名无关，签名类型相关。方法多一个或少一个都不匹配，返回具体类型也不能替代接口返回类型——Go 没有返回值协变。

普通运行时 interface 的方法集还可通过嵌入合并。带类型集合的约束接口主要用于泛型，见 [[31-方法约束]] 与 [[32-类型集合波浪号与联合约束]]。

## 2. 可独立运行的最小 demo：包含 typed nil

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import "fmt"

type Namer interface {
	Name() string
}

type User struct {
	name string
}

func (u *User) Name() string {
	if u == nil {
		return "<nil-user>"
	}
	return u.name
}

var _ Namer = (*User)(nil)

func main() {
	var empty Namer
	var user *User
	var typedNil Namer = user

	fmt.Println(empty == nil)
	fmt.Println(typedNil == nil)
	fmt.Println(typedNil.Name())
}
```

预期输出 `true`、`false`、`<nil-user>`。最后一行之所以不 panic，是方法显式处理 nil receiver；多数方法不会。示例已在 `go1.26.0` 验证。

## 3. 编译期实现断言

```go
var _ Namer = (*User)(nil)
```

左边 `_` 不保存值，右边也不分配对象；编译器只检查 `*User` 能否赋给 `Namer`。常见形式：

```go
var _ I = T{}          // 明确要求值类型实现
var _ I = (*T)(nil)    // 明确要求指针类型实现
```

断言适合框架 adapter、跨 package 契约和生成 fake 的接口；无需给每个局部小类型机械添加。

## 4. 常用设计模式

### 模式一：consumer-defined interface

需要依赖的一方声明最小方法集，producer 不必预先预测所有用途。这降低耦合，也是 [[25-小接口与依赖注入]] 的主线。

### 模式二：adapter 边界断言

项目类型适配外部框架时，在实现旁放断言，使依赖升级造成的签名变化尽早变成编译错误。

### 模式三：能力组合

从多个小接口组合大接口，只在调用方确实需要全部能力时使用。不要把实现类型所有方法都复制进接口。

### 模式四：零实现与 fake

小接口容易写内存 fake 或函数 adapter，测试无需启动真实网络/Kubernetes。接口的价值来自替换边界，不是“所有 struct 前都加一层”。

## 5. NGF 实例：Reconciler 适配 controller-runtime

`internal/framework/controller/reconciler.go` 有：

```go
var _ reconcile.Reconciler = &Reconciler{}
```

`*Reconciler` 的 `Reconcile(context.Context, reconcile.Request) (reconcile.Result, error)` 与 controller-runtime 接口匹配。路径是：

1. `Register` 组装 `ReconcilerConfig`；
2. `cfg.newReconciler(recCfg)` 返回 `*Reconciler`；
3. builder 的 `Complete` 接收框架接口；
4. 运行时框架按接口调用 `Reconcile`；
5. 项目方法把 Kubernetes 读取结果转换为内部 event。

断言不注册 controller，也不触发运行时检查；它只证明方法集。真正的构造/注册发生在 `internal/framework/controller/register.go:Register`。`internal/framework/controller/reconciler_test.go` 用 fake Getter 与 channel 检查实际行为。

仓库中另一个清晰例子是 `internal/framework/runnables/runnables.go`：`Leader` 同时断言 `manager.Runnable` 和 `manager.LeaderElectionRunnable`，前者由嵌入提升，后者由新增方法组成。

## 6. nil 接口陷阱

接口值只有动态类型和动态值都为空时才 `== nil`。函数返回具体 typed nil 时尤其危险：

```go
func bad() Namer {
	var u *User
	return u // 返回的接口非 nil
}
```

更安全的做法是失败/缺失时显式 `return nil`；或者不要用 nil 表达业务缺失，返回 `(Namer, bool)` / `(Namer, error)`。

> [!warning] 反射式 nil 检查不是通用补丁
> `reflect.ValueOf(x).IsNil()` 只适用于部分 kind，还会让 API 语义更模糊。优先修正返回边界。

## 7. 边界与误区

- 接口不是对象的完整副本；它持有动态值，若是指针则共享对象；
- 接口越大，生产者越难实现、测试 fake 越重；
- 接口参数不应为了“未来可能”提前扩张；
- 接口调用有动态分派，但通常先优化设计清晰度，再用 benchmark 判断开销；
- 接口不能表达构造不变量，仍需构造函数或校验。

## 8. 练习与检查点

1. 删除 demo 的 nil receiver 分支。检查最后一行运行时 panic，而 `typedNil == nil` 仍为 false。
2. 把 `Name` 改成值 receiver。检查 `(*User)(nil)` 仍在方法集中，但通过 nil pointer 调用提升的值方法会 panic。
3. 给 `Namer` 增加 `Rename(string)`，观察断言使缺失立刻编译失败。反思 consumer 是否真的需要它。
