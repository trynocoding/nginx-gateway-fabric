# 36 `%w` 与错误链

> [!abstract]
> `%w` 同时保留人类可读的阶段语境和机器可判断的底层原因。包装后的错误不是原错误本身，但通过 `Unwrap` 链，`errors.Is`/`errors.As` 仍能识别原因。是否包装也是 API 契约选择。

## 学习目标与前置

- 区分 `%v` 与 `%w`；
- 理解 `Unwrap() error` 链和逐层包装；
- 设计“动作 + 对象 + 原因”的错误语境；
- 读懂 NGF 文件写入路径的包装与延迟 Close 聚合。

前置：[[17-多返回值与error-last]]、[[20-defer与资源清理]]。

## 1. error 是值，错误链是约定

最小接口只有一个方法：

```go
type error interface {
	Error() string
}
```

包装错误通常额外实现下面这个接口形状。

**说明性示例：**

```go
type unwrapper interface {
	Unwrap() error
}
```

`errors.Unwrap(err)` 只取一层；反复调用形成单链。`fmt.Errorf("load config: %w", err)` 会创建实现该约定的包装值。

```go
fmt.Errorf("load config: %v", err) // 只拼文本，原因身份丢失
fmt.Errorf("load config: %w", err) // 文本 + 可遍历原因
```

## 2. 可独立运行的最小 demo

**可运行 demo（Go 1.26.0，保存为 `main.go`）：**

```go
package main

import (
	"errors"
	"fmt"
)

var ErrMissing = errors.New("missing")

func readConfig(name string) error {
	return fmt.Errorf("read %q: %w", name, ErrMissing)
}

func start() error {
	if err := readConfig("gateway.yaml"); err != nil {
		return fmt.Errorf("start controller: %w", err)
	}
	return nil
}

func main() {
	err := start()
	fmt.Println(err)
	fmt.Println(errors.Is(err, ErrMissing))
	for level := 0; err != nil; level++ {
		fmt.Printf("%d: %v\n", level, err)
		err = errors.Unwrap(err)
	}
}
```

验证要点：输出包含两层语境，`errors.Is` 为 true，手工循环依次看到外层、read 层、哨兵层。

```bash
gofmt -w main.go
go run main.go
```

## 3. 常用包装模式

### 3.1 跨抽象层补动作

底层返回 `permission denied`，文件层补 `create "/path"`，启动层再补 `write deployment context`。每层只增加自己知道的信息。

### 3.2 保留哨兵分类

上层需要判断 not found、EOF、canceled 等语义时必须 `%w`，不能依赖错误字符串。

### 3.3 自定义错误继续包装原因

带字段的错误类型可以同时实现 `Unwrap() error`，让 `errors.As` 取结构、`errors.Is` 查根因。见 [[37-errors-Is-As与自定义错误]]。

### 3.4 多个原因

一个操作有多个独立失败时用 `errors.Join`，其约定是 `Unwrap() []error`，不是单链。见 [[38-errors-Join多错误聚合]]。

## 4. NGF：`file.Write` 的错误上下文

**NGF 缩写源码（不是独立 demo）：**

```go
f, err := fileMgr.Create(file.Path)
if err != nil {
	return fmt.Errorf("failed to create file %q: %w", file.Path, err)
}

if err := fileMgr.Write(f, file.Content); err != nil {
	resultErr = fmt.Errorf("failed to write file %q: %w", file.Path, err)
	return resultErr
}
```

数据/错误关系：生成的 `file.File` → OSFileManager Create/Chmod/Write → 每个阶段补动作、路径、权限 → 调用者得到可诊断文本，同时仍能判断 fake/OS 返回的根因。

Close 在 defer 中执行，可能与主体失败同时发生：

```go
resultErr = errors.Join(resultErr,
	fmt.Errorf("failed to close file %q: %w", file.Path, err))
```

这说明“单原因包装”和“多原因聚合”可以嵌套：Join 的某个子节点本身仍有 `%w` 单链。

`internal/framework/file/file_test.go` 的 `Edge cases with IO errors` 分别注入 Create/Chmod/Write 错误，并要求调用返回错误且与注入错误匹配；正常路径还验证权限与内容。调用方 `cmd/gateway/initialize.go` 会再包装为 `failed to write deployment context file`，形成跨层链。

## 5. `%w` 的边界与误区

- `err == ErrMissing` 只比较当前值；包装后通常为 false，应用 `errors.Is`。
- `%w` 的对象必须是 error；错误格式会产生异常格式文本，不是正常错误处理。
- Go 允许一个 `fmt.Errorf` 中多个 `%w`，结果是多原因错误；为可读控制流，独立错误常显式用 `errors.Join`。
- 不要每层重复“failed to”；更有价值的是动作、对象和关键标识。
- 不要把敏感 token、证书或完整私有 URL 填进错误语境。
- 包装某个外部错误等于允许调用者依赖其 `Is/As` 语义；公共 API 迁移时需考虑兼容性。

## 6. 错误策略选择

| 情况 | 策略 |
|---|---|
| 能恢复/分类的运行错误 | 返回并 `%w` 包装 |
| 只需展示、明确不暴露底层身份 | `%v` 或新错误，但记录契约 |
| 多个操作都应尝试 | 收集后 `errors.Join` |
| 内部不变量破坏 | 有充分证据时 panic |

NGF 文件 I/O 来自运行环境，权限不足、磁盘失败都可预期，因此返回 error；未知内部 File.Type 才使用 panic。两种边界不可互换。

## 7. 练习与答案

1. 将 demo 中 `%w` 改成 `%v`，观察 `errors.Is`。
2. 写一个 `OperationError`，含 `Op`、`Path`、`Err` 并实现 `Unwrap`。
3. 为什么只比较 `err.Error()` 不足以验证根因？

检查点：

```go
type OperationError struct {
	Op   string
	Path string
	Err  error
}

func (e *OperationError) Error() string {
	return fmt.Sprintf("%s %q: %v", e.Op, e.Path, e.Err)
}

func (e *OperationError) Unwrap() error {
	return e.Err
}
```

第 3 题：文本会随新增语境变化，也可能有相同文本的不同错误；`Is/As` 使用语义身份或类型合同。
