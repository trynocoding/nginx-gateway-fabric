# 51 io 接口与资源所有权

> [!abstract] 核心结论
> `io.Reader`/`Writer` 抽象字节流而非文件。调用者必须先处理 `n` 个有效字节，再处理 error；`io.Copy` 负责循环与短写检查，但不替你 Close。通常“谁 Open/Create，谁 Close”，借用的 Reader/Writer 则不能擅自关闭。

## 学习目标与前置

前置：[[15-string-byte与rune]]、[[20-defer与资源清理]]、[[25-小接口与依赖注入]]。完成后应能：

- 精确解释 `Read(p) (n,error)`、`Write(p) (n,error)`；
- 正确处理 `n>0` 与 `io.EOF` 同时出现、短读和短写；
- 使用 `io.Copy/CopyN/ReadAll/ReadFull` 并选择合适边界；
- 读懂 NGF `copyFile → OSFileManager.Copy → io.Copy` 的所有权分工。

## 1. 两个最小接口

```go
type Reader interface {
	Read(p []byte) (n int, err error)
}

type Writer interface {
	Write(p []byte) (n int, err error)
}
```

Reader 可返回 `n>0, err==io.EOF`，也可返回比缓冲区小的 n 而 err=nil；短读不是错误。Writer 应返回已接受字节数，若 `n<len(p)` 就必须给非 nil error。调用方不能假设一次调用完成全部 I/O。

## 2. 可独立运行 demo

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import (
	"bytes"
	"errors"
	"fmt"
	"io"
	"strings"
)

type finalReader struct{ done bool }

func (r *finalReader) Read(p []byte) (int, error) {
	if r.done {
		return 0, io.EOF
	}
	r.done = true
	return copy(p, "NGF"), io.EOF
}

type shortWriter struct{}

func (shortWriter) Write(p []byte) (int, error) { return len(p) - 1, nil }

func main() {
	buf := make([]byte, 8)
	n, err := (&finalReader{}).Read(buf)
	fmt.Printf("data=%q eof=%v\n", buf[:n], errors.Is(err, io.EOF))

	var dst bytes.Buffer
	written, err := io.Copy(&dst, strings.NewReader("gateway"))
	fmt.Println(written, dst.String(), err)

	_, err = io.Copy(shortWriter{}, strings.NewReader("abc"))
	fmt.Println(errors.Is(err, io.ErrShortWrite))
}
```

预期输出证明三件事：即使收到 EOF 仍有 `"NGF"`；Copy 返回 7 和完整内容；违反 Writer 短写契约会转成 `io.ErrShortWrite`。已在 `go1.26.0` 执行验证。

## 3. 正确的读取循环

通用顺序是：

```go
for {
	n, err := r.Read(buf)
	if n > 0 {
		consume(buf[:n])
	}
	if err == io.EOF {
		break
	}
	if err != nil {
		return err
	}
}
```

**说明性示例。** 若先判断 EOF 再消费 n，会丢掉最后一批数据。也要防止实现反复返回 `(0,nil)` 导致忙循环；标准 Reader 约定调用方应谨慎处理这种情况。

## 4. 常用组合器

| API | 适用问题 | 关键边界 |
|---|---|---|
| `io.Copy(dst,src)` | 流式复制到 EOF | 返回字节数；不 Close |
| `io.CopyN` | 只复制 n 字节 | 提前 EOF 返回 error |
| `io.ReadAll` | 数据确定较小 | 全量驻留内存 |
| `io.ReadFull` | 缓冲区必须填满 | 可能 `ErrUnexpectedEOF` |
| `io.LimitReader` | 限制不可信输入 | 只限制读取视图，不报“超限” |
| `io.MultiReader/Writer` | 顺序拼接/广播写 | 错误传播规则需查看 API |
| `io.TeeReader` | 读取同时旁路写 | 写失败表现为读错误 |

`io.Copy` 可能优先调用 source 的 `WriterTo` 或 destination 的 `ReaderFrom`，因此不是固定分配一个缓冲区的实现细节。依赖应面向行为，不应假设具体 Read/Write 次数。

## 5. Closer 所有权

`io.Reader` 不承诺有 Close；文件/网络响应常额外实现 `io.ReadCloser`。基本规则：

- 函数自己 Open/Create：函数负责 Close；
- 参数只是借入 Reader/Writer：默认调用方负责 Close；
- 函数返回 ReadCloser：返回值接收者负责 Close；
- wrapper 是否传播 Close 要在类型/API 中明确。

Close 也可能失败，尤其写文件 flush 时。只读文件 Close 错误通常次要，但不能把“忽略 Close error”变成所有资源的模板。

## 6. NGF 数据路径

`cmd/gateway/initialize.go:copyFile` 的顺序是：

1. `OSFileManager.Open(src)`，成功后立即 defer source Close；
2. `Create(dest/base(src))`，成功后立即 defer destination Close；
3. `OSFileManager.Copy(destFile, srcFile)`；
4. `Chmod` 为 regular file mode；
5. 任一步 error 都加阶段上下文返回。

`internal/framework/file/os_filemanager.go:(*StdLibOSFileManager).Copy` 只是：

```go
_, err := io.Copy(dst, src)
return err
```

它只借用两个接口，不 Open 也不 Close；所有权留在 `copyFile`。它丢弃 byte count，因为调用方只需要“全部复制到 EOF 或失败”的结果。若业务要显示进度/验证大小，接口应返回 `(int64,error)`，而不是重新 stat 猜测。

## 7. 失败与测试边界

`cmd/gateway/initialize_test.go:TestCopyFile` 使用真实临时文件验证内容与 mode；`TestCopyFileErrors` 用生成 fake 覆盖 Open/Create/Copy/Chmod 失败。`TestInitialize_OSS` 还检查每个配置文件都会 Open/Create/Copy。

当前 `copyFile` 用 `defer srcFile.Close()` 和 `defer destFile.Close()`，没有合并 Close error。对初始化复制的 regular files，这是当前明确实现；若目标内容必须在 Close 才保证持久化，就需像 `internal/framework/file/file.go:Write` 那样聚合 Close error。

`StdLibOSFileManager.Copy` 本身没有专门单测；其核心行为属于标准库 `io.Copy`，项目测试主要验证 adapter 被正确调用和错误传播。

## 8. 常见误区与迁移边界

> [!warning] interface 越小，所有权越不能靠猜
> `io.Reader` 看不出是否要 Close。把所有权写在构造/函数注释和返回类型中。

- 不要把 EOF 当“读取失败”；它是流正常结束哨兵；
- 不要假定一次 File.Write 一定完成，遵守 Writer 契约或用 `io.Copy`/辅助循环；
- 不可信大输入不能直接 ReadAll；先定大小和超限错误；
- Copy 成功不等于目标文件已 fsync 到磁盘，持久性是另一层契约。

**直接迁移：** 谁打开谁关闭；先处理 n 再处理 err。**条件迁移：** 忽略 Close error，只适用于明确的只读/非关键路径。**不要复制：** helper 擅自关闭借入 Reader。

## 9. 练习与检查点
