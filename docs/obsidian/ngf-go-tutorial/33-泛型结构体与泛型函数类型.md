# 33 泛型结构体与泛型函数类型

> [!abstract]
> 泛型结构体把“同一种容器形状”参数化，实例化后的字段仍是强类型；泛型函数类型把回调的输入/输出与容器元素绑定。两者组合适合分页响应、缓存和处理流水线。

## 学习目标与前置

- 声明、实例化并为泛型结构体定义方法；
- 读懂 `Page[User]` 与 `Page[Policy]` 是不同的具体类型；
- 用命名泛型函数类型表达回调合同；
- 追踪 NGF `n1cPagedResult[T]` 的 JSON 解码与分页搜索。

前置：[[07-struct-tag与JSON字段语义]]、[[19-函数值闭包与高阶函数]]、[[29-泛型函数与类型推断]]。

## 1. 泛型类型的基本语法

```go
type Page[T any] struct {
	Items []T
	Total int
}
```

`Page` 本身不是可直接声明变量的完整类型，必须实例化：

```go
var users Page[User]
var names Page[string]
// var p Page // 编译失败：缺少类型实参
```

方法接收者要重新声明类型形参，但不重复约束：

```go
func (p Page[T]) Empty() bool { return len(p.Items) == 0 }
```

`Page[User]` 的 `Items` 是 `[]User`，不会因为共用容器而退化为 `[]any`。

## 2. 泛型函数类型

函数类型也可参数化：

```go
type Predicate[T any] func(T) bool
type Mapper[S, D any] func(S) D
```

它们让字段或参数的关系可读：一个 `Predicate[User]` 只能接收 `User`。命名函数类型适合跨多个签名重复使用或承载领域含义；只使用一次时，直接写 `func(T) bool` 更清楚。

## 3. 可独立运行的最小 demo

**可运行 demo（Go 1.26.0，保存为 `main.go`）：**

```go
package main

import (
	"encoding/json"
	"fmt"
)

type Page[T any] struct {
	Items []T `json:"items"`
	Total int `json:"total"`
}

type Predicate[T any] func(T) bool

func (p Page[T]) Find(match Predicate[T]) (T, bool) {
	for _, item := range p.Items {
		if match(item) {
			return item, true
		}
	}
	var zero T
	return zero, false
}

type Policy struct {
	Name string `json:"name"`
}

func main() {
	data := []byte(`{"items":[{"name":"edge"}],"total":1}`)
	var page Page[Policy]
	if err := json.Unmarshal(data, &page); err != nil {
		panic(err)
	}
	p, ok := page.Find(func(p Policy) bool { return p.Name == "edge" })
	fmt.Println(page.Total, p.Name, ok)
}
```

验证：

```bash
gofmt -w main.go
go run main.go
# 1 edge true
```

## 4. 常用模式

### 4.1 协议 envelope

分页 JSON 的 `items` 形状相同、元素形状不同，最适合 `Page[T]`。序列化库从实例化类型获得 `T` 的字段元数据。

### 4.2 强类型 cache/result

`Result[T]` 或 `Cache[K, V]` 能保留返回值类型，避免每次读取都断言。若不同结果还需要统一存放，再在外层增加非泛型接口。

### 4.3 策略字段

结构体可存 `Predicate[T]`、`func(T) error` 等策略，让数据类型与策略输入严格一致。

### 4.4 方法返回零值

`Find` 找不到元素时要 `var zero T`。若“不存在”是常态，用 `(T, bool)`；若需要语境和传播，用 `(T, error)`。

## 5. NGF：N1C 分页响应

**NGF 缩写源码（不是独立 demo）：**

```go
type n1cPagedResult[T any] struct {
	Items []T `json:"items"`
	Total int  `json:"total"`
}

var resp n1cPagedResult[T]
if err := json.Unmarshal(body, &resp); err != nil {
	var zero T
	return zero, fmt.Errorf("failed to parse %s response: %w", errMsg, err)
}
```

数据关系：N1C list HTTP body → `json.Unmarshal` 根据当前 `T` 解码 `items` → `paginatedSearch` 遍历 `[]T` → 调用 typed predicate → 返回具体条目。`findN1CLogProfile` 实例化为 `n1cLogProfileItem`，其他查找实例化为各自 policy/version item。

这里不使用 `[]map[string]any`，因为 object ID、name、latest 等字段会失去编译期检查；也不需要反射手工构造页面类型，`encoding/json` 已在库边界处理元数据。

`internal/framework/waf/fetch/fetch_test.go:TestN1CFetchPolicyByNamePagination` 用 `httptest.Server` 验证目标位于第二页时的 offset/limit 和最终 bundle；`TestN1CFetchPolicyNotFound` 验证耗尽页面的错误路径。

## 6. 边界与误区

- `Page[A]` 不能直接赋值给 `Page[B]`，即使 A、B 底层结构相同。
- JSON 的运行时字段错误仍由 `Unmarshal` 返回 error；泛型不验证业务值。
- `T any` 不意味着可以对 item 调领域方法；需要方法约束。
- 不要为只出现一次的回调机械创建命名函数类型；命名应带来复用或语义。
- 协议 envelope 可泛化，协议之间不同的分页终止规则不应被同一结构体掩盖。

## 7. 选择与迁移边界

选择泛型结构体，因为字段布局重复且元素类型在编译期已知；选择函数值，因为“如何匹配”随调用者变化。若服务端未来按 cursor 分页，`Total` 和 offset 终止不再成立，应新建 cursor envelope/算法，而不是给当前类型堆可选字段。

若所有 item 都必须提供 `ObjectID()`，可以收紧为方法约束；若只有某个调用需要该行为，把逻辑留在 predicate，避免污染通用 envelope。

## 8. 练习与答案

1. 为 demo 增加 `MapPage[S, D]`，保留 Total 并变换 Items。
2. 构造非法 JSON，确认错误来自哪里。
3. 为什么 `[]Page[any]{page}` 不能接收 `Page[Policy]`？

检查点：

```go
func MapPage[S, D any](p Page[S], f func(S) D) Page[D] {
	out := Page[D]{Total: p.Total, Items: make([]D, len(p.Items))}
	for i, item := range p.Items {
		out.Items[i] = f(item)
	}
	return out
}
```

第 3 题：Go 泛型不是协变类型系统，两个实例化是不同具体类型；显式逐项转换才能说明语义。
