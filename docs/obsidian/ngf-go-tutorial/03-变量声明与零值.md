# 03 变量声明与零值

> [!abstract]
> Go 保证每个变量在使用前都有值：未显式初始化时得到该类型的零值。好的类型会让零值表达安全、常见的初始状态；不能依赖零值的资源则由构造函数建立。

## 学习目标与前置

读完后应能：

- 使用 `var`、短声明 `:=`、赋值 `=` 和多变量声明；
- 列出基本类型、指针、slice、map、channel、struct 的零值；
- 判断 nil slice 为什么可读/可 append，而 nil map、nil channel 有何不同；
- 解释 NGF readiness 对 `bool` 零值和 `readyCh` 构造的不同选择。

前置：[[02-package导出规则与internal边界]]。指针和 nil 的完整语义见 [[08-指针nil与可选字段]]。

## 1. 四种声明/赋值写法

```go
var count int          // 声明，count == 0
var name = "gateway"  // 从右侧推断 string
ready := false         // 短声明，只能用于函数体
count = 1              // 给已有变量赋值，不是声明
```

短声明左边至少要有一个当前作用域中的新变量：

```go
value, err := load()
value, err = loadAgain()
```

第二行不能继续用 `:=`，因为没有新变量。多返回值与 `error` 在 [[17-多返回值与error-last]] 展开。

### 零值表

| 类型 | 零值 | 是否通常可直接用 |
|---|---|---|
| `bool` | `false` | 是 |
| 整数/浮点 | `0` | 是，但要确认 0 有业务意义 |
| `string` | `""` | 是 |
| 指针、函数、接口 | `nil` | 只能比较/检查，调用前需证明有效 |
| slice | `nil` | 可 `len`、`range`、`append` |
| map | `nil` | 可读，写入会 panic |
| channel | `nil` | 收发永久阻塞，`close` 会 panic |
| struct | 各字段零值 | 取决于字段组合是否满足不变量 |

> [!warning]
> “有零值”不等于“零值可完成所有操作”。语言保证初始化，不保证业务不变量已经建立。

## 2. 可独立运行的最小 demo（Go 1.26）

```go
// 可运行示例；Go 1.26.0
package main

import "fmt"

type State struct {
	ready bool
	items []string
	index map[string]int
}

func main() {
	var s State
	fmt.Println(s.ready, len(s.items), s.index == nil)

	// nil slice 可以 append；append 返回更新后的 slice header。
	s.items = append(s.items, "gateway")

	// nil map 必须先 make 才能写。
	s.index = make(map[string]int)
	s.index["gateway"] = 1

	fmt.Println(s.items, s.index["gateway"])
}
```

验证：

```bash
gofmt -w main.go
go run main.go
# false 0 true
# [gateway] 1
```

失败实验：删除 `make(map[string]int)` 后运行，写 nil map 会触发 `panic: assignment to entry in nil map`。这是语言错误，不应靠 `recover` 变成正常分支。

## 3. 常用模式

### 3.1 让零值代表“尚未发生”

计数器的 `0`、状态位的 `false`、空字符串通常无需构造。字段越少需要手工初始化，类型越不容易被错误构造。

### 3.2 使用 `var zero T` 得到泛型零值

```go
func First[T any](xs []T) (T, bool) {
	if len(xs) == 0 {
		var zero T
		return zero, false
	}
	return xs[0], true
}
```

不要写 `return nil, false`，因为 `T` 可能是 `int`。

### 3.3 构造函数初始化资源字段

map、必须收发的 channel、互相关联的字段应在 `NewXxx` 中初始化，使调用者拿到即可用的值。

### 3.4 用延迟初始化避免不必要成本

nil slice 可以在首次 append 时分配。只有能清楚区分“未初始化”和“初始化为空”且并发安全时，才对 map 或重资源做 lazy init。

## 4. NGF：readiness 状态如何利用零值

焦点：`ngf:internal/controller/health.go:graphBuiltHealthChecker`。

**NGF 缩写源码（不是独立 demo）：**

```go
type graphBuiltHealthChecker struct {
	readyCh chan struct{}
	lock    sync.RWMutex
	ready   bool
}

func newGraphBuiltHealthChecker() *graphBuiltHealthChecker {
	return &graphBuiltHealthChecker{readyCh: make(chan struct{})}
}
```

这里三个字段展示三种零值策略：

| 字段 | 初始来源 | 为什么 |
|---|---|---|
| `ready bool` | 零值 `false` | 正好表示“初始 graph 尚未构建” |
| `lock sync.RWMutex` | Mutex 可用零值 | 无需构造，直接加锁 |
| `readyCh chan struct{}` | `make` | nil channel 无法用关闭事件广播 readiness |

运行关系：

```text
newGraphBuiltHealthChecker
  → ready=false，readyCh 已分配
  → readyCheck 读锁下观察 ready
  → 初次 graph 构建后 setAsReady
  → 写锁下 ready=true，并 close(readyCh) 广播
```

`readyCheck` 在 `ready == false` 时返回错误；`setAsReady` 置 true 并关闭 channel；`getReadyCh` 只向外暴露接收方向。`internal/controller/health_test.go:TestReadyCheck` 佐证未就绪与就绪两个状态。

### 为什么不能直接使用 struct 零值

`var h graphBuiltHealthChecker` 的 `readyCheck` 尚可返回未就绪，但 `readyCh` 为 nil：等待它会永远阻塞，`setAsReady` 中 `close(h.readyCh)` 会 panic。因此构造函数不是样板代码，而是补齐 bool/Mutex 零值无法建立的 channel 不变量。

另一个边界是 `setAsReady` 只应执行一次；第二次关闭同一 channel 会 panic。当前类型依赖调用生命周期保证这一点，而不是用 `sync.Once` 自行防重。迁移到调用次数不受控的组件时不要照搬。

## 5. 设计与迁移判断

- **直接复用**：让 `false` 表达“尚未 ready”，让 Mutex 使用零值。
- **条件复用**：用 close(channel) 广播一次性状态，前提是存在唯一关闭者且只关闭一次。
- **不要照搬**：对可重复切换的状态使用一次性关闭 channel；它无法重新打开。
- **建议**：若类型的零值不可用，应提供构造函数并在注释/测试中固定不变量。

## 6. 练习与检查点

1. nil slice 与空但非 nil slice 的 `len` 都是多少？都是 0；JSON 等边界上二者可能编码不同。
2. `var m map[string]int; fmt.Println(m["x"])` 是否 panic？不会，读得零值 0；写才 panic。
3. 为什么 `sync.RWMutex` 不需要 `new`？它的零值就是未锁定、可用状态。
4. 如何让 `setAsReady` 可安全重复调用？可在类型内部用 `sync.Once` 包住状态转换与 close，但需重新评估锁和测试。
