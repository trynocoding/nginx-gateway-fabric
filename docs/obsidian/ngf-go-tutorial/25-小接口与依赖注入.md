# 25 小接口与依赖注入

> [!abstract] 核心结论
> 依赖注入不是框架专属技术：把依赖作为构造参数传入即可。接口应由 consumer 按“它实际调用的方法”定义；方法越少，替换、测试和演进越容易。具体类型作为返回值、接口作为输入通常是良好起点，而非绝对规则。

## 学习目标与前置

前置：[[24-接口的隐式实现]]、[[28-构造函数与不变量]] 可交叉阅读。完成后应能：

- 在 consumer package 定义 1–3 个方法的小接口；
- 用构造注入、函数注入和配置 struct 注入依赖；
- 编写手工 fake，验证成功与失败路径；
- 读懂 NGF telemetry collector 如何依赖 GraphGetter、ConfigurationGetter 与 Kubernetes Reader。

## 1. 为什么“由使用者定义”

假设 producer `Database` 有二十个方法，而报表组件只调用 `Lookup`。若报表依赖 producer 定义的二十方法接口，fake 也必须实现全部；producer 新增方法还会无故破坏 consumer。由报表组件定义：

```go
type Lookup interface {
	Lookup(string) (string, error)
}
```

任何已有或未来类型只要有这一方法就能注入，producer 不需要导入 consumer。

## 2. 可独立运行的最小 demo

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import (
	"errors"
	"fmt"
)

type Getter interface {
	Get(string) (string, error)
}

var errNotFound = errors.New("not found")

type Service struct {
	getter Getter
}

func NewService(getter Getter) (*Service, error) {
	if getter == nil {
		return nil, errors.New("getter is required")
	}
	return &Service{getter: getter}, nil
}

func (s *Service) Greeting(name string) (string, error) {
	value, err := s.getter.Get(name)
	if err != nil {
		return "", fmt.Errorf("load greeting: %w", err)
	}
	return value + ", " + name, nil
}

type mapGetter map[string]string

func (m mapGetter) Get(key string) (string, error) {
	value, ok := m[key]
	if !ok {
		return "", errNotFound
	}
	return value, nil
}

func main() {
	service, _ := NewService(mapGetter{"Ada": "hello"})
	message, err := service.Greeting("Ada")
	fmt.Println(message, err)
	_, err = service.Greeting("Grace")
	fmt.Println(errors.Is(err, errNotFound))
}
```

预期输出 `hello, Ada <nil>` 和 `true`。第二行证明包装后的错误仍可用 `errors.Is` 识别 sentinel。demo 已在 `go1.26.0` 执行验证。

## 3. 常用注入模式

### 模式一：构造函数参数

依赖在对象整个生命周期都必需时，用 `NewService(getter Getter)`。构造时拒绝 nil，之后方法可假设不变量成立。

### 模式二：配置 struct

依赖多且都有名字时，用 `Config{Reader, Logger, Clock}`，避免长位置参数。配置 struct 并不自动校验，构造函数仍应建立不变量。

### 模式三：函数类型

只有一个行为且无需多实现方法时，可注入 `type Now func() time.Time`。测试传闭包即可，见 [[19-函数值闭包与高阶函数]]。

### 模式四：窄 adapter

外部依赖接口过大时，在 consumer 边界包一层只暴露所需能力。adapter 负责参数/错误模型转换，业务层不直接依赖 SDK 全貌。

## 4. NGF 生产实例：telemetry collector

`internal/controller/telemetry/collector.go` 在 consumer package 内定义：

```go
type GraphGetter interface {
	GetLatestGraph() *graph.Graph
}

type ConfigurationGetter interface {
	GetLatestConfiguration() []*dataplane.Configuration
}
```

`DataCollectorConfig` 再把这两个接口、`client.Reader` 以及纯数据配置集中传给 `NewDataCollectorImpl`。`Collect` 的数据关系为：

1. `GraphGetter` 提供最新 graph；nil graph 立即返回 error；
2. `ConfigurationGetter` 提供派生的 dataplane configurations；
3. `K8sClientReader` 只负责读 cluster/pod/ReplicaSet 等观测数据；
4. collector 合成新的 telemetry `Data`，不接管这些依赖的生命周期。

两个本地接口各只有一个方法，恰好反映 consumer 所需。Kubernetes `client.Reader` 是上游接口，包含 Get/List；collector 两者都使用，所以没有再人为缩小为两个重复接口。

`internal/controller/telemetry/collector_test.go` 使用 Counterfeiter 生成的 `FakeGraphGetter`、`FakeConfigurationGetter`、`FakeReader` 注入正常值和错误，覆盖 nil graph、读取失败与正常聚合。测试无需真实集群，这正是依赖倒置带来的收益。

## 5. 不变量与 nil 边界

当前 `NewDataCollectorImpl` 直接保存 cfg，没有逐项 nil 校验；`Collect` 会调用其中接口。因此调用方 wiring 是构造前置条件，测试通过完整 fake 满足它。迁移到另一个项目时，不应机械复制“构造函数不校验”：

- 必需依赖且缺失属于配置错误：构造函数返回 error；
- 依赖在编译期/内部 wiring 中已绝对保证：可用注释与集中组装维护，但 panic 位置会更晚；
- 可选依赖：用 nil 分支明确降级行为，而不是任由方法调用 panic。

还要防 typed nil：`getter != nil` 不能排除接口内部装着 `(*Fake)(nil)`。API 应尽量让构造调用传真实对象；若这确实是外部输入边界，再设计显式验证。

## 6. 常见误区与迁移边界

> [!warning] 不要为 mock 而接口化一切
> 纯数据 struct、稳定无副作用算法通常直接依赖具体类型更简单。接口应出现在需要替换的行为边界。

- producer-owned “万能 Repository” 会把无关 consumer 耦合在一起；
- 返回接口会隐藏额外能力与构造结果，除非确实需要替换实现或封装；
- service locator / 全局变量也能“拿到依赖”，但把真实输入隐藏了，不利测试与推理；
- fake 只能证明调用契约，不能替代真实 Kubernetes/API 集成测试。

**直接迁移：** consumer-owned 小接口和构造注入。**条件迁移：** 生成 fake，需同步 `go generate` 与生成代码边界。**不建议：** 仅为测试访问私有字段而造大接口。

## 7. 练习与检查点

1. 把 demo 的 sentinel 改成导出的 `ErrNotFound`，思考何时值得把错误变成 API 契约。
2. 给 `Service` 再加 `Logger`，比较位置参数和 Config struct。检查字段名是否让 wiring 更清晰。
3. 浏览 `DataCollectorConfig`，为每个依赖标注“必需/可选/纯数据”。检查源码是否真的实现了你假设的 nil 行为。
