# NGF 源码中的 Go 1.26 语法教程

> [!abstract] 这套教程解决什么问题
> 目标不是让你“见过”某个 Go 语法，而是让你能解释它的运行语义，能识别工程中的常用模式，能在 NGF
> 源码里找到真实证据，并能把同一思路安全地迁移到后续项目。

源码基线：`nginx-gateway-fabric@918d0fa7`；语言与本地验证工具链：Go `1.26.0`。

## 每章怎么学

每篇正文都围绕同一个学习闭环展开，但会根据主题调整章节，而不是机械套模板：

1. **先建立语法模型**：解释声明方式、零值、类型关系或并发契约，不预设读者已经会用。
2. **运行最小 demo**：示例应能独立复制；正文会注明它是“可运行示例”“说明性片段”还是“源码节选”。
3. **比较常用模式**：不仅写正确写法，也解释替代方案、适用边界和容易误判的情况。
4. **回到 NGF 源码**：从生产符号、调用关系、状态所有权和测试说明项目为什么这样写。
5. **做迁移练习**：先预测，再修改，再运行验证命令，最后用检查点复盘自己的理解。

> [!tip] 推荐动作
> 阅读代码前先手写或运行 demo；看完 NGF 实例后，合上正文，用自己的话回答“谁创建、谁拥有、谁修改、
> 谁负责失败与清理”。回答不出来，就说明还没有真正吃透。

## 三条阅读路线

### 路线 A：从零系统学习

按 `01 → 65` 顺序阅读。前四部分建立语言、抽象、错误与并发基础；第五部分连接标准库和代码生成；
第六部分把前面的能力放进 Kubernetes Controller 的真实工作流。

### 路线 B：已有 Go 基础，直接读工程

先读 [[24-接口的隐式实现]]、[[25-小接口与依赖注入]]、[[36-错误链与百分号w]]、
[[44-context传递与取消]]、[[49-EventLoop批处理与状态所有权]]，再进入 `57–65`。遇到语法盲点时回跳到对应基础章。

### 路线 C：围绕一次配置变更追源码

按下面的路径阅读：

```text
Kubernetes 事件
  → 60 Reconciler 契约
  → 61 Predicate 过滤
  → 62 Cache / FieldIndexer
  → 63 Graph 领域建模
  → 64 ChangeProcessor
  → 65 Dataplane / NGINX 配置
```

并发控制和取消语义不清楚时，配合 `40–49` 阅读；测试替身与行为验证不清楚时，配合 `57–59` 阅读。

## 示例与证据约定

| 标记 | 含义 | 可以如何使用 |
|---|---|---|
| 可运行示例 | 独立的教学程序或测试，正文给出工具链与命令 | 复制到临时目录运行并修改 |
| 说明性片段 | 为突出一个语义而省略外围代码 | 不应直接当成完整程序 |
| NGF 源码节选 | 当前基线中的最小必要代码 | 用给出的路径和符号回到仓库核对 |
| 源码事实 | 可由当前实现直接证明 | 可作为理解当前版本的依据 |
| 测试佐证 | 有测试明确约束该行为 | 修改前应定位并运行相关测试 |
| 推断/建议 | 对设计理由或迁移方式的分析 | 需要结合自己的约束重新判断 |

为避免每个短语法片段都重复标注，整个 corpus 采用以下默认规则：**未在局部明确标为“可运行示例”或
“NGF 源码节选”的 Go 代码围栏，一律视为说明性示例，可能省略外围声明；`text` 围栏是伪代码/流程；
`mermaid` 围栏是结构或时序图；紧跟可运行示例的 `bash` 围栏是验证命令。** 局部标签优先于这条默认规则。

教程只承诺源码基线 `918d0fa7` 上的结论。代码演进后，先查 [[99-源码索引与术语表]] 中的符号，再核对当前实现和测试。

## 一章学完的判定标准

不要以“读到末尾”作为完成。至少能做到下面四件事再进入下一章：

- 不看正文写出最小 demo，并预测正常与失败输入的结果；
- 指出 NGF 示例的创建者、调用者、状态所有者和清理/错误责任；
- 解释项目写法与 demo 相比多了哪些生产约束；
- 把模式迁移到一个新问题，并说明哪部分不能直接照搬。
