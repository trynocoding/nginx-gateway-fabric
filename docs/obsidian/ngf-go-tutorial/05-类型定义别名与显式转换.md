# 05 类型定义、类型别名与显式转换

> [!abstract]
> `type UserID string` 创建新命名类型，获得独立类型身份；`type ID = string` 只是同一类型的别名。新命名类型与底层类型通常要显式转换，这一点正是领域边界的保护。

## 学习目标与前置

- 看懂类型定义与别名的语法差异；
- 区分底层类型相同与类型相同；
- 知道何时可以显式转换、转换会不会复制或改变表示；
- 解释 NGF 为何把 Kubernetes `ServiceType` 再包装为 API 类型。

前置：[[04-有类型常量与枚举模式]]。

## 1. 两种 `type` 声明

```go
type Port int32       // 定义新的命名类型
type PortAlias = int32 // int32 的别名
```

`Port` 与 `int32` 类型不同，虽然底层类型相同；`PortAlias` 与 `int32` 完全相同。别名主要服务迁移、重导出和生成代码兼容，不是给业务概念“起好听的名字”。

```go
var raw int32 = 8080
var p Port = Port(raw)       // 必须显式转换
var a PortAlias = raw        // 无需转换
```

### 命名类型能拥有方法

```go
func (p Port) Valid() bool { return p > 0 && p <= 65535 }
```

不能在非本包定义的类型上新增方法；先定义本包命名类型，就能围绕领域语义建立方法集。

### 转换不是解析

`Port(8080)` 是数值类型转换；`int("8080")` 不合法，因为字符串到整数需要 `strconv.Atoi/ParseInt`。转换只在语言允许的表示关系间进行，不负责输入校验。

## 2. 可独立运行的最小 demo（Go 1.26）

```go
// 可运行示例；Go 1.26.0
package main

import "fmt"

type ServiceType string
type ServiceTypeAlias = string

const LoadBalancer ServiceType = "LoadBalancer"

func describe(t ServiceType) string {
	return "service=" + string(t)
}

func main() {
	raw := "LoadBalancer"
	typed := ServiceType(raw)
	var alias ServiceTypeAlias = raw

	fmt.Println(describe(typed))
	fmt.Printf("typed=%T alias=%T equal-text=%v\n", typed, alias, string(typed) == alias)
}
```

```bash
gofmt -w main.go
go run main.go
# service=LoadBalancer
# typed=main.ServiceType alias=string equal-text=true
```

失败例：把 `describe(typed)` 改成 `describe(raw)`，编译器会拒绝把变量 `string` 隐式当成 `ServiceType`。显式写 `ServiceType(raw)` 同时也是审查者可见的边界。

## 3. 常用模式

| 模式 | 写法 | 价值 | 风险 |
|---|---|---|---|
| 领域标识 | `type RouteID string` | 不同 ID 不会误传 | 仍需校验内容 |
| 单位类型 | `type Milliseconds int64` | 防止单位混用 | 转换处要明确 |
| 外部类型适配 | `type ServiceType corev1.ServiceType` | API 层独立声明标记/方法 | 上下游值域需同步 |
| 兼容别名 | `type OldName = NewName` | 迁移期间不破坏类型身份 | 无法形成新边界 |

### 何时用 struct 而不是命名基础类型

当不变量需要多个字段共同表达，或值不能由任意基础值直接构造时，用未导出字段的 struct + 构造函数更强。`type Email string` 仍允许 `Email("bad")`。

## 4. NGF：API `ServiceType` 到 Kubernetes 类型

焦点：`ngf:apis/v1alpha2/nginxproxy_types.go:ServiceType`。

**NGF 缩写源码（不是独立 demo）：**

```go
type ServiceType corev1.ServiceType

const (
	ServiceTypeClusterIP ServiceType = ServiceType(corev1.ServiceTypeClusterIP)
	ServiceTypeNodePort ServiceType = ServiceType(corev1.ServiceTypeNodePort)
	ServiceTypeLoadBalancer ServiceType = ServiceType(corev1.ServiceTypeLoadBalancer)
)
```

这是类型定义，不是别名。因此 NGF 可以在自己的 CRD 字段上挂 `+kubebuilder:validation:Enum=...`，并控制对外暴露的值集合，同时复用 Kubernetes 类型的 string 表示。

生产转换在 `ngf:internal/controller/provisioner/objects.go:NginxProvisioner.buildNginxService`：

```go
serviceType = corev1.ServiceType(*serviceCfg.ServiceType)
```

数据关系：

```text
NginxProxy CRD 的 *v1alpha2.ServiceType
  → nil 时采用 defaultServiceType
  → 非 nil 时显式转换为 corev1.ServiceType
  → 写入 corev1.Service.Spec.Type
```

显式转换很重要：它标出 API 模型跨入 Kubernetes 对象模型的位置。随后逻辑用 `corev1.ServiceTypeClusterIP` 判断 NodePort 和外部流量策略，这时已经处在 Kubernetes 类型域内。

`internal/controller/provisioner/objects_test.go` 的 Service 构造场景可用于确认配置到对象的结果；类型转换本身由编译器保证，但允许值集合仍依赖 CRD 校验和业务测试。

## 5. 为什么不用别名

若写 `type ServiceType = corev1.ServiceType`，NGF API 字段与 Kubernetes 类型将完全同一：迁移容易，却弱化了“这是 NGF CRD 契约”的所有权。当前定义允许 NGF 独立附加生成标记和命名常量。

代价是上游新增 Service 类型时不会自动成为 NGF 的支持值。维护者需要明确选择是否同步；这通常是 API 边界所希望的行为。

## 6. 边界与误区

- 底层类型相同不意味着可隐式赋值；变量通常要显式转换；
- 转换不会自动校验枚举合法性；`ServiceType("Bad")` 仍可构造；
- 数值窄化可能截断，如 `uint8(256) == 0`，边界前先校验范围；
- `string([]byte)` 会复制/转换数据语义，不要把所有转换都理解成“零成本改名”；
- 别名没有独立方法集和类型身份，不适合表达需要隔离的领域概念。

## 7. 练习与答案

1. `type A string; type B string`，能否把变量 A 直接赋给 B？不能，要 `B(a)`。
2. `type A = string` 呢？A 与 string 相同，可直接赋值。
3. 为什么转换后仍要校验 Service 类型？转换只改变静态类型，不限制字符串值域。
4. 若上游新增值，NGF 会自动支持吗？不会；需同步 API Enum、常量、转换后的业务逻辑与测试。
