# 59 Counterfeiter、fake 与可测试依赖注入

> [!abstract]
> Counterfeiter 从接口生成可编程 fake：测试可设置 stub/默认返回/按调用次序返回，并检查次数和参数。生成器只是放大已有边界；真正的可测试性来自窄接口、构造注入和明确的副作用所有权。

## 学习目标与前置

- 从 `go:generate` 和 `counterfeiter:generate` 追到生成文件；
- 使用 `XStub`、`XReturns`、`XReturnsOnCall`、`XCallCount`、`XArgsForCall`；
- 判断生成 fake 的锁能保证什么、不能保证什么；
- 区分生成文件、手写接口和测试场景的修改责任。

前置：[[24-接口的隐式实现]]、[[25-小接口与依赖注入]]、[[56-go-generate与生成代码边界]]。

## 1. 最小接口与生成指令

```go
//go:generate go tool counterfeiter -generate

//counterfeiter:generate . Sender
type Sender interface {
	Send(context.Context, []byte) error
}
```

两条注释职责不同：

- `//go:generate ...`：Go 工具实际执行的命令；
- `//counterfeiter:generate . Sender`：Counterfeiter 扫描的生成规格；`.` 表示当前包；
- `go tool counterfeiter`：使用 go.mod `tool` 管理的工具版本，而非 PATH 上未知二进制。

NGF `go.mod` 固定 `github.com/maxbrunsfeld/counterfeiter/v6 v6.12.2` 并声明 tool；生成命令因此可复现。

## 2. 生成 fake 提供什么

以方法 `Send(ctx, req) error` 为例，常见 API：

| 生成成员 | 用途 |
|---|---|
| `SendStub func(...) error` | 每次调用运行自定义逻辑 |
| `SendCalls(stub)` | 线程安全设置 stub 的 helper |
| `SendReturns(err)` | 设置默认返回 |
| `SendReturnsOnCall(i, err)` | 第 i 次调用特定返回，i 从 0 开始 |
| `SendCallCount()` | 已调用次数 |
| `SendArgsForCall(i)` | 取第 i 次参数 |
| `Invocations()` | 跨方法调用记录（不一定表达严格全局时序） |

fake 方法通常先加锁记录参数/复制配置，解锁后才调用 stub，避免 stub 回调 fake 时死锁。

## 3. 可独立运行的最小测试（Go 1.26）

下面手写一个与生成 API 等价的最小 fake，用来先吃透机制；真实项目应由 Counterfeiter 生成而非手抄。

```go
// 可运行测试；Go 1.26.0
package sender

import (
	"context"
	"errors"
	"sync"
	"testing"
)

type Sender interface {
	Send(context.Context, string) error
}

type service struct{ sender Sender }

func (s service) Deliver(ctx context.Context, msg string) error {
	return s.sender.Send(ctx, msg)
}

type FakeSender struct {
	mu         sync.Mutex
	calls      []string
	SendStub   func(context.Context, string) error
	defaultErr error
}

func (f *FakeSender) Send(ctx context.Context, msg string) error {
	f.mu.Lock()
	f.calls = append(f.calls, msg)
	stub, ret := f.SendStub, f.defaultErr
	f.mu.Unlock()
	if stub != nil {
		return stub(ctx, msg)
	}
	return ret
}

func (f *FakeSender) SendReturns(err error) {
	f.mu.Lock()
	defer f.mu.Unlock()
	f.SendStub = nil
	f.defaultErr = err
}

func (f *FakeSender) SendCallCount() int {
	f.mu.Lock()
	defer f.mu.Unlock()
	return len(f.calls)
}

func TestDeliver(t *testing.T) {
	fake := &FakeSender{}
	fake.SendReturns(errors.New("offline"))

	err := (service{sender: fake}).Deliver(t.Context(), "config")
	if !errors.Is(err, fake.defaultErr) {
		t.Fatalf("got %v", err)
	}
	if got := fake.SendCallCount(); got != 1 {
		t.Fatalf("calls=%d, want 1", got)
	}
}
```

```bash
gofmt -w sender_test.go
go test -race -v .
```

手写版刻意很小；它没有 Counterfeiter 的按调用返回、参数 getter 和 invocation map。其价值是说明 fake 只是一个实现接口、记录调用的普通 Go 类型。

## 4. NGF 的 Messenger 生成链

接口：`ngf:internal/controller/nginx/agent/grpc/messenger/messenger.go:Messenger`。

```go
// NGF 缩写源码，不是 demo
//go:generate go tool counterfeiter -generate
//counterfeiter:generate . Messenger
type Messenger interface {
	Run(context.Context)
	Send(context.Context, *pb.ManagementPlaneRequest) error
	Messages() <-chan *pb.DataPlaneResponse
	Errors() <-chan error
}
```

生成物：`messengerfakes/fake_messenger.go:FakeMessenger`，文件首行明确 `Code generated by counterfeiter. DO NOT EDIT.`，末尾有：

```go
var _ messenger.Messenger = new(FakeMessenger)
```

生成链：

```text
手写 Messenger 接口/指令
  → go generate / Counterfeiter v6.12.2
  → FakeMessenger（生成）
  → command_test 配置 fake
  → commandService 只依赖 Messenger 接口
```

生产实现 `NginxAgentMessenger` 包装 gRPC stream；构造函数 `New` 返回 `Messenger`。`commandService.Subscribe` 生产路径调用 `messenger.New(in)`，而 `setInitialConfig(..., msgr Messenger)` 接收接口参数，测试可以直接注入 FakeMessenger。

## 5. NGF 测试里的四种编程方式

`internal/controller/nginx/agent/command_test.go:TestSetInitialConfig_Errors` 展示：

### 默认返回

```go
msgr.SendReturns(errors.New("send error"))
```

所有未单独指定的 Send 调用返回此错误。

### 按调用序号返回

```go
msgr.SendReturnsOnCall(1, errors.New("api send error"))
```

第 0 次初始配置成功，第 1 次 Plus API 请求失败，精确模拟阶段性故障。

### channel 返回

`MessagesReturns(msgCh)` / `ErrorsReturns(errCh)` 把异步结果所有权交给测试；buffer=1 避免 setup 阶段发送阻塞。

### 调用断言

其他测试通过 `SendCallCount`、`SendArgsForCall` 确认调用次数和请求内容。次数断言应与可观察行为结合；只断言“调用过”容易把实现细节锁死。

## 6. 生成 fake 的并发语义

当前 FakeMessenger 每个方法有 `sync.RWMutex`，记录参数、读取 ReturnsOnCall、设置 stub 和 CallCount 都加锁；`Invocations` 另有锁。这使 fake 自身的记录结构在并发调用下避免明显 data race。

但它不保证：

- 你的 stub 闭包访问外部 map/slice 是线程安全的；
- 返回的 channel 被正确关闭、没有 goroutine 泄漏；
- `ArgsForCall` 返回的指针参数之后未被生产代码修改；
- 两个不同方法之间的业务顺序满足协议；
- 测试不会在调用进行中重配 Returns 而产生语义竞态。

因此仍要 `go test -race`，并用 channel/Eventually 明确同步点。

## 7. 生成代码边界与安全修改

需要改方法签名时：

1. 修改手写接口和生产实现；
2. 运行仓库约定的生成命令；
3. 检查生成 diff，只接受对应 fake 变化；
4. 修复测试的 Returns/ArgsForCall；
5. 运行相关包测试与生成代码一致性检查。

不要直接改 `fake_messenger.go`，下次生成会覆盖。也不要为了测试巨型对象而定义巨型接口；接口应由消费者需要的方法决定。

## 8. fake、stub、mock 的术语边界

项目文件名使用 fake：它有可编程行为并记录调用。不同社区对 fake/mock/stub 定义不完全一致，维护时以具体能力而非名称沟通：

- stub：提供预设结果；
- spy：记录调用；
- mock：测试验证交互期望；
- fake：可工作的简化实现或生成的多能力替身。

Counterfeiter 生成物同时具备 stub 与 spy 能力。

## 9. 练习与检查点

1. `ReturnsOnCall(1, err)` 对哪次调用生效？第二次，索引从 0 开始。
2. fake 有锁是否可共享给并行子测试？不建议；锁防 race，但调用记录和配置仍互相污染。
3. 新增 Messenger 方法后该改生成文件吗？不直接改；改接口/实现后重新生成。
4. 何时手写 fake 更好？需要具有领域行为的内存实现、生成 API 反而更复杂时；仍应实现窄接口。
