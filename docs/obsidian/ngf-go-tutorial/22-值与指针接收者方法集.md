# 22 值与指针接收者方法集

> [!abstract] 核心结论
> 定义类型 `T` 的方法集只含接收者为 `T` 的方法；`*T` 的方法集同时含接收者为 `T` 和 `*T` 的方法。编译器对可寻址值的直接调用会自动取址，但接口赋值按真实方法集检查，不做这层便利转换。

## 学习目标与前置

前置：[[21-方法与接收者]]、[[08-指针nil与可选字段]]。完成后应能：

- 精确写出 `T` 与 `*T` 的方法集；
- 解释 `v.M()` 能调用却 `var i I = v` 编译失败的原因；
- 按修改语义、复制成本、identity、锁与一致性选择 receiver；
- 对比 NGF 的值接收者 `DataCollectorImpl.Collect` 和指针接收者 `Reconciler.Reconcile`。

## 1. 方法集规则

设：

```go
func (T) Read() {}
func (*T) Write() {}
```

则：

| 表达式/类型 | `Read` | `Write` |
|---|---:|---:|
| `T` 的方法集 | 是 | 否 |
| `*T` 的方法集 | 是 | 是 |
| 可寻址变量 `v T` 直接调用 | 是 | 是，编译器改写为 `(&v).Write()` |
| 非可寻址临时值 | 是 | 通常不能调用指针方法 |

接口实现检查使用前两行，不使用“直接调用”的语法糖。

## 2. 可独立运行的最小 demo

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import "fmt"

type Reader interface {
	Read() int
}

type ReadWriter interface {
	Read() int
	Write(int)
}

type Cell struct {
	value int
}

func (c Cell) Read() int    { return c.value }
func (c *Cell) Write(v int) { c.value = v }

var _ Reader = Cell{}
var _ Reader = (*Cell)(nil)
var _ ReadWriter = (*Cell)(nil)

func main() {
	cell := Cell{value: 1}
	cell.Write(2) // 自动取址只发生在这种可寻址直接调用

	var reader Reader = cell
	var readWriter ReadWriter = &cell
	fmt.Println(reader.Read(), readWriter.Read())
}
```

运行预期输出 `2 2`。若取消注释 `var _ ReadWriter = Cell{}`，编译器会指出 `Cell` 缺 `Write`。示例已用 `go1.26.0` 验证成功路径和该编译失败路径。

## 3. receiver 选择矩阵

| 条件 | 更常选值 | 更常选指针 |
|---|---|---|
| 方法需要修改接收者 | 否 | 是 |
| struct 很小且语义像值 | 是 | 可选 |
| 含 map/slice 指针式字段 | 仍可修改底层数据，需谨慎 | 常用，语义更明确 |
| 含 Mutex/Cond/原子字段 | 不应复制 | 是 |
| 需要稳定 identity 或允许 nil | 否 | 是 |
| 同类型其他方法已用指针 | 保持一致优先 | 是 |

值接收者复制的是 struct 字段；map、slice、pointer 字段复制后仍可能指向同一底层状态。因此“值接收者=不可变”并不成立。

## 4. NGF 的两种选择

### 值接收者：`DataCollectorImpl.Collect`

`internal/controller/telemetry/collector.go:DataCollectorImpl` 只含 `cfg DataCollectorConfig`，其 `Collect(ctx)` 使用值接收者。它读取 cfg 中的 getter/client，再构建新的 `Data` 返回；方法不替换 `c.cfg`。

`internal/controller/telemetry/job_worker.go:DataCollector` 是只有 `Collect` 的 consumer interface。由于值接收者的方法同时属于 `DataCollectorImpl` 与 `*DataCollectorImpl` 的方法集，两种都可满足接口。实际构造函数 `NewDataCollectorImpl` 返回指针，说明“能用值满足接口”不等于项目必须复制它。

### 指针接收者：`Reconciler.Reconcile`

`internal/framework/controller/reconciler.go` 的 `Reconcile`、`mustCreateNewObject` 都是 `*Reconciler` 方法，并有：

```go
var _ reconcile.Reconciler = &Reconciler{}
```

因此 `*Reconciler` 实现框架接口，`Reconciler` 值不实现。`NewReconciler` 和注册路径也都返回/传递指针，保持 identity 一致。

## 5. 常见失败例

> [!warning] “能点出来”不等于“满足接口”
> `cell.Write(2)` 成功是因为变量可寻址；`var rw ReadWriter = cell` 失败是因为 `Cell` 的方法集没有 `Write`。

- 从 map 取出的 struct 值不可寻址，不能直接调用其指针 receiver；可存 `*T` 或取出、修改、写回；
- 混用 receiver 会让接口满足关系难以预测，除非确有值/指针语义差异；
- 已发布类型从值接收者改为指针接收者，可能使原本用 `T` 满足的接口失效，是兼容性变化；
- 编译期断言用 `(*T)(nil)` 不会分配对象，只在编译期检查方法集。

## 6. 迁移边界

**直接复用：** 小、只读、值语义类型用 value receiver；可变、含锁、需 identity 类型统一 pointer receiver。**条件复用：** 即使构造函数返回指针，也可因 value receiver 让 `T`/`*T` 均满足只读接口。**不要复制：** 仅凭“方法没改字段”就把含锁或大 struct 改成值接收者。

## 7. 练习与检查点

1. 给 demo 增加 `map[string]Cell`，尝试 `m["x"].Write(3)`。检查编译失败，再改成 `map[string]*Cell`。
2. 把 `Read` 改为指针接收者。检查 `Reader = Cell{}` 的断言也失败。
3. 查看 `DataCollectorImpl` 的 cfg 字段，判断复制是否深复制。答案：不是；接口、slice/map/pointer 字段仍共享动态值或底层对象。
