# 23 嵌入和组合

> [!abstract] 核心结论
> struct 嵌入省略字段名，并把内层字段/方法提升为便捷选择器；它仍是组合，不是继承。外层可覆盖提升的方法，但不存在虚方法派发。接口嵌入则把方法集合并为更大的契约。

## 学习目标与前置

前置：[[06-struct与复合字面量]]、[[21-方法与接收者]]、[[22-值与指针接收者方法集]]。完成后应能：

- 声明匿名嵌入字段并显式访问内层值；
- 解释字段/方法提升、覆盖和歧义；
- 区分嵌入具体 struct、指针和 interface 的零值风险；
- 读懂 NGF `Leader` 如何装饰 `manager.Runnable`，而不是继承它。

## 1. struct 嵌入的真实模型

```go
type Outer struct {
	Inner
}
```

`Outer` 确实有一个字段，字段名就是类型名 `Inner`。`outer.Run()` 只是 `outer.Inner.Run()` 的提升写法。复合字面量不能把提升字段当外层字段直接设置，必须写 `Outer{Inner: Inner{...}}`。

若外层自己定义 `Run`，选择 `outer.Run()` 时优先外层；仍可显式 `outer.Inner.Run()`。若同一层嵌入的两个字段都提供同名成员，未限定选择会编译歧义。

## 2. 可独立运行的最小 demo

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import "fmt"

type Runner interface {
	Run() string
}

type Task struct {
	Name string
}

func (t Task) Run() string { return "run " + t.Name }

type LeaderTask struct {
	Runner
}

func (LeaderTask) NeedLeader() bool { return true }

func main() {
	leader := LeaderTask{Runner: Task{Name: "reload"}}
	fmt.Println(leader.Run(), leader.NeedLeader())
	fmt.Println(leader.Runner.Run())
}
```

运行预期输出 `run reload true`，随后 `run reload`。`LeaderTask.Run` 来自嵌入 interface 的提升；显式字段仍可访问。已在 `go1.26.0` 验证。

## 3. 三种常用模式

### 模式一：能力装饰器

外层嵌入一个 interface，复用其方法，再增加元数据或行为。优点是包装任意实现；风险是 nil interface 零值在提升方法调用时 panic。

### 模式二：数据分组

外层嵌入稳定的小 struct，减少字段选择器层级。若只为少打几个字符而嵌入大型类型，会把过多 API 暴露到外层。

### 模式三：接口组合

```go
type ReadWriter interface {
	Reader
	Writer
}
```

这是方法集合并，不携带字段和实现。小接口通常由 consumer 定义；只在 consumer 真需要两组能力时再组合。

## 4. NGF 主实例：leader runnable 装饰器

`internal/framework/runnables/runnables.go`：

```go
type Leader struct {
	manager.Runnable
}

func (r *Leader) NeedLeaderElection() bool { return true }
```

**生产源码节选。** `manager.Runnable` 提供被提升的 `Start(context.Context) error`；外层 `Leader` 新增 `NeedLeaderElection`，从而满足 `manager.LeaderElectionRunnable`。编译期断言同时检查两个接口。

调用关系不是基类 override：

1. 构造端把一个具体 runnable 放入 `Leader{Runnable: x}`；
2. manager 看到外层既可 `Start`，又可询问 `NeedLeaderElection`；
3. `Start` 实际委托给动态的嵌入 interface；
4. `NeedLeaderElection` 固定返回 true。

`LeaderOrNonLeader` 使用相同组合但返回 false。`internal/framework/runnables/runnables_test.go` 验证两者选举需求与委托行为。

### 另一个实例：生成器要求的结构嵌入

`internal/controller/telemetry/collector.go:Data` 嵌入 `tel.Data` 和 `NGFResourceCounts`。注释明确这是 telemetry generator 所需。这里嵌入影响生成协议字段布局，不能只为“代码更短”随意改成命名字段；修改需同步运行生成器并检查 generated attributes/tests。

## 5. 零值、方法集和失败边界

> [!warning] 嵌入 interface 的零值通常不可调用
> `var leader Leader; leader.Start(ctx)` 会通过 nil 动态接口调用并 panic。构造点必须填入 `Runnable`。

- 嵌入 `*Inner` 的零值也是 nil，提升方法是否安全取决于方法体；
- 外层方法调用内层方法时不会自动动态回调外层 override；Go 没有继承式虚派发；
- 提升的方法会影响外层方法集，进而意外满足接口；公共 API 改嵌入类型可能破坏兼容；
- JSON/YAML 对匿名字段有自己的 flatten/冲突规则，不能仅凭 Go 选择器推断序列化结果。

## 6. 迁移边界

**直接复用：** 用嵌入 interface 做薄装饰器。**有条件复用：** 数据 struct 嵌入，需确认序列化和生成器契约。**不要复制：** 把嵌入当继承，或通过嵌入暴露一个本不属于外层职责的大 API。

## 7. 练习与检查点

1. 给 demo 的 `LeaderTask` 定义自己的 `Run`。检查 `leader.Run()` 与 `leader.Runner.Run()` 输出不同。
2. 令 `var leader LeaderTask` 并调用 `Run`。检查运行时 panic，说明零值不完整。
3. 再嵌入一个也有 `Run` 的 interface。检查未限定 `leader.Run()` 编译歧义。
