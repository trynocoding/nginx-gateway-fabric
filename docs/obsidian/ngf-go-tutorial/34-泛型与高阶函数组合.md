# 34 泛型和高阶函数组合

> [!abstract]
> 泛型固定“数据是什么类型”，高阶函数注入“这次怎样处理”。组合后可以复用遍历、分页、错误包装等骨架，同时避免 `any`、类型断言和重复控制流。

## 学习目标与前置

- 把变化点提取为 `func` 参数，把稳定控制流留在泛型函数；
- 设计 predicate、mapper、URL builder 的签名与错误边界；
- 理解闭包捕获的参数如何进入通用算法；
- 追踪 NGF `paginatedSearch[T]` 的构建、解码、匹配和终止。

前置：[[19-函数值闭包与高阶函数]]、[[29-泛型函数与类型推断]]、[[33-泛型结构体与泛型函数类型]]。

## 1. 两个维度分别负责什么

```go
func Find[T any](items []T, match func(T) bool) (T, bool)
```

- `T` 保证输入元素和返回元素一致；
- `match` 让每次调用选择不同规则；
- `Find` 只拥有遍历和提前返回。

若没有泛型，常见退化是 `[]any` + 断言；若没有高阶函数，常见退化是为 name、ID、status 各复制一份循环。

## 2. 可独立运行的最小 demo

**可运行 demo（Go 1.26.0，保存为 `main.go`）：**

```go
package main

import (
	"fmt"
	"strings"
)

func Find[T any](items []T, match func(T) bool) (T, bool) {
	for _, item := range items {
		if match(item) {
			return item, true
		}
	}
	var zero T
	return zero, false
}

func Map[S, D any](items []S, f func(S) (D, error)) ([]D, error) {
	out := make([]D, 0, len(items))
	for i, item := range items {
		v, err := f(item)
		if err != nil {
			return nil, fmt.Errorf("item %d: %w", i, err)
		}
		out = append(out, v)
	}
	return out, nil
}

func main() {
	word, ok := Find([]string{"gateway", "fabric"}, func(s string) bool {
		return strings.HasPrefix(s, "fab")
	})
	lengths, err := Map([]string{"ngf", "go"}, func(s string) (int, error) {
		return len(s), nil
	})
	fmt.Println(word, ok, lengths, err)
}
```

验证：

```bash
gofmt -w main.go
go run main.go
# fabric true [3 2] <nil>
```

## 3. 常用组合模式

### 3.1 predicate：选择

`func(T) bool` 适合无失败的本地判断。若判断会 I/O 或解析，应改为 `func(T) (bool, error)`，不能把错误伪装成 false。

### 3.2 mapper：变换

`func(S) (D, error)` 允许输入输出类型不同。通用骨架负责下标语境和提前退出，策略负责单元素转换。

### 3.3 builder：延迟构造

`func(offset, limit int) (string, error)` 让分页器拥有页序和终止条件，调用者拥有具体 endpoint 规则。

### 3.4 闭包注入环境

predicate 可捕获 `targetName`，builder 可捕获 base URL/namespace。应只读捕获值；循环中并发闭包若共享可变变量，需要额外同步和清晰所有权。

## 4. NGF：`paginatedSearch`

**NGF 缩写源码（不是独立 demo）：**

```go
func paginatedSearch[T any](
	ctx context.Context,
	client *http.Client,
	auth *BundleAuth,
	buildURL func(offset, limit int) (string, error),
	pred func(item T) (result T, found bool),
	errMsg string,
) (T, error)
```

稳定骨架：从 offset=1 开始 → builder 生成 URL → `doGet` → 解码 `n1cPagedResult[T]` → 遍历并调用 pred → 命中立即返回 → 根据 `Total` 终止 → 未找到返回带语境的 error。

调用者 `findN1CLogProfile` 注入：

- builder 捕获 base URL 和 namespace；
- predicate 捕获 profileName，并返回匹配 item；
- 类型推断从两个闭包得到 `T=n1cLogProfileItem`。

错误策略有清晰所有权：builder、HTTP、JSON 错误由骨架用 `%w` 加阶段语境；“是否匹配”不失败，所以 predicate 不返回 error；耗尽是领域失败而非 panic。

`internal/framework/waf/fetch/fetch_test.go:TestN1CFetchPolicyByNamePagination` 验证 offset 1/101 跨页、认证头和最终效果；`TestN1CFetchPolicyNotFound` 验证耗尽分支。这些是高阶抽象没有吞掉协议细节的证据。

## 5. 边界和失败例

- callback 为 nil 时调用会 panic；若来自外部调用者，构造时应校验；私有函数可把非 nil 视为接线不变量。
- predicate 若有副作用，分页重试或重复 item 会放大副作用；最好保持纯函数。
- 把 `context.Context` 隐藏在闭包里会模糊取消链；I/O 骨架应显式接收并传递 ctx。
- 过度抽象会产生十几个 callback。变化点若并不独立，两个具体函数往往更可读。
- 泛型不会自动并发。是否并发、是否保持顺序、失败是否取消都必须由骨架明确。

## 6. 选择与迁移边界

此处选择泛型而非接口，因为 item 没有共同必需行为；选择函数值而非反射，因为变化点是行为，不是字段元数据；选择返回 error 而非 panic，因为网络、JSON 和未找到都是预期运行失败。

若分页协议改为 cursor，稳定骨架已经变化，应新建 cursor 搜索器。若 predicate 开始返回不同结果类型，可改为两个类型参数 `T, R`，不要把结果塞进 `any`。

## 7. 练习与答案

1. 把 `Find` 改为 `FindE[T]`，predicate 返回 `(bool, error)`。
2. 给 `Map` 的转换函数制造错误，确认错误链能被 `errors.Is` 判断。
3. 解释为什么 `paginatedSearch` 的 pred 返回 `(T, bool)` 而非仅 `bool`。

检查点：

```go
func FindE[T any](items []T, match func(T) (bool, error)) (T, bool, error) {
	for _, item := range items {
		ok, err := match(item)
		if err != nil {
			var zero T
			return zero, false, err
		}
		if ok {
			return item, true, nil
		}
	}
	var zero T
	return zero, false, nil
}
```

第 3 题：调用者可能想返回 item 的规范化/选择结果；当前实现允许 predicate 同时选择并返回结果，但仍限制结果为同一个 T。若真实需求出现 `T→R`，再引入第二类型参数。
