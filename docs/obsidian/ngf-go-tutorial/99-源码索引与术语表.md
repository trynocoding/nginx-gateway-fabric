## 如何核对一条教程结论

以“`Reconciler` 会把删除转换成 `DeleteEvent`”为例：

1. 用 `codegraph node Reconciler.Reconcile` 查看当前实现、调用者和测试；
2. 在 `internal/framework/controller/reconciler.go` 中确认 `IsNotFound → obj=nil → DeleteEvent` 分支；
3. 打开 `internal/framework/controller/reconciler_test.go`，确认测试覆盖的输入与输出；
4. 再沿 `EventCh` 查消费者，避免把“事件已入队”误写成“NGINX 已更新”；
5. 若源码与教程不一致，以当前源码和测试为准，并把教程的 `status` 改成 `stale` 后再修订。

这套方法同样适用于语言特性：先在最小 demo 中证明语义，再检查 NGF 是否额外附加了所有权、校验、
框架回调或生成代码边界。

## 核心术语

| 术语 | 本教程中的精确定义 | 容易混淆的概念 |
|---|---|---|
| 语言版本 | `go.mod` 的 `go` 指令选择的 Go 语言与命令语义基线 | 当前 PATH 中 `go` 二进制的版本 |
| 零值 | 类型在未显式初始化时得到的合法初始值 | 业务上“有效的默认配置” |
| 命名类型 | 通过类型定义创建、拥有独立类型身份和方法集的类型 | 类型别名 |
| 底层类型 | 决定表示和部分可用操作的类型结构 | 两个命名类型是否可直接赋值 |
| 方法集 | 某个类型或指针类型可用于接口实现的方法集合 | 源文件中“看起来属于它”的所有方法 |
| 小接口 | 按消费者真实需要定义、方法数量少的行为契约 | 把实现类型所有方法都复制进接口 |
| 泛型约束 | 类型形参允许的类型集合和操作权限 | 运行时类型判断 |
| 错误链 | 通过 `Unwrap`、`%w` 或多错误展开关系形成的可遍历结构 | 仅拼接后的错误字符串 |
| goroutine 所有权 | 谁启动、如何停止、谁等待以及如何报告失败的责任集合 | 只写一个 `go` 关键字 |
| channel 所有权 | 谁创建、谁发送、谁关闭，以及取消时如何解除阻塞 | 每个持有 channel 的调用者都可关闭它 |
| 背压 | 下游处理能力通过阻塞、容量或拒绝策略反向限制上游 | 单纯使用缓冲 channel |
| Context | 沿调用链携带截止时间、取消信号和请求级值的接口 | 可选参数包或长期业务数据容器 |
| 临界区 | 为保持共享状态不变量而必须互斥执行的最小代码范围 | 整个函数都加锁 |
| 幂等 | 同一有效输入重复处理不会产生额外的可观察副作用 | 函数绝对不执行工作 |
| Reconcile | controller-runtime 针对一个 key 调用的“使观察状态趋向期望状态”的回调 | 一次回调就完成整个系统收敛 |
| Predicate | 进入工作队列前对资源事件做的过滤 | Reconcile 内的业务校验 |
| Cache | controller-runtime/client-go 维护的本地对象视图 | Kubernetes API Server 的权威状态 |
| FieldIndexer | 为缓存对象建立字段到对象集合的反向索引 | 数据库里的持久化索引 |
| Graph | NGF 从 Kubernetes 对象派生出的领域关系与状态模型 | API 对象的简单复制 |
| Dataplane Configuration | 与具体 NGINX 文本模板解耦的中间配置模型 | 最终 `nginx.conf` 文件 |
| 生成代码边界 | 由工具从接口/模式生成、应通过源定义和命令更新的代码区域 | 可以手工长期维护的普通源码 |

## 证据等级

- **源码事实**：当前 revision 的实现直接支持该结论。
- **测试佐证**：命名测试明确约束了行为；测试存在不代表覆盖所有失败路径。
- **运行观察**：在记录的工具链和命令下实际得到的结果，只对该环境负责。
- **推断**：证据支持但源码没有直接声明的设计解释，必须保留替代解释。
- **建议**：迁移到其他项目时的工程选择，不等于 NGF 维护者意图。

## 从源码继续深挖

- 想理解包和 API 边界：从 `cmd/gateway/` 的入口，沿构造函数进入 `internal/controller/`。
- 想理解事件控制面：从 `internal/framework/controller/Reconciler.Reconcile` 追到 `internal/framework/events/`。
- 想理解状态转换：从 `internal/controller/state/graph.BuildGraph` 追到 `state/dataplane.BuildConfiguration`。
- 想理解配置落地：从 `internal/controller/nginx/config.GeneratorImpl.Generate` 继续追踪 agent 文件消息。
- 想安全修改：先找到拥有该结论的教程章，再运行其证据表中的聚焦测试，最后扩大到 `make unit-test`。
