# 57 表驱动测试、子测试与并行测试

> [!abstract]
> `testing.T` 是单个测试节点的生命周期句柄；表驱动把案例数据化，`t.Run` 建立可单独选择的子测试，`t.Parallel` 只应用于状态真正隔离的节点。`TempDir`、`Context`、`Helper`、`Cleanup` 则负责临时资源、取消传播、错误归因和清理顺序。

## 学习目标与前置

- 写完整 `*_test.go`、表格案例和子测试；
- 理解父测试与并行子测试何时暂停/继续；
- 正确处理循环变量、共享 fake、环境变量与临时文件；
- 使用 `t.Helper()`、`t.TempDir()`、`t.Context()`、`t.Cleanup()`；
- 从 NGF 测试判断哪些案例可并行、哪些资源必须每例创建。

## 1. `testing.T` 是什么

Go 命令查找名字以 `Test` 开头、签名为 `func(*testing.T)` 的函数：

```go
func TestAdd(t *testing.T) {
	if got := 1 + 1; got != 2 {
		t.Fatalf("got %d, want 2", got)
	}
}
```

常见方法：

| 方法 | 作用 | 是否继续当前测试 goroutine |
|---|---|---|
| `t.Error/Errorf` | 记录失败 | 继续 |
| `t.Fatal/Fatalf` | 记录失败并结束当前测试 | 不继续 |
| `t.Run(name, fn)` | 创建子测试 | 返回子测试是否成功 |
| `t.Skip` | 标记跳过 | 结束当前测试 |
| `t.Log/Logf` | 记录诊断 | 继续 |

`Fatal` 只终止调用它的测试 goroutine，不能从任意后台 goroutine 调用来控制父测试；后台结果应通过 channel 返回给测试 goroutine。

## 2. 表驱动与 `t.Run`

```go
tests := []struct {
	name    string
	input   string
	want    int
	wantErr bool
}{
	{name: "valid", input: "42", want: 42},
	{name: "empty", input: "", wantErr: true},
}

for _, tc := range tests {
	t.Run(tc.name, func(t *testing.T) {
		// Arrange → Act → Assert
	})
}
```

案例名是测试 API 的一部分：`go test -run 'TestParse/empty'` 可精确复现。把输入、期望错误、setup 和 wantCalls 等真正变化的数据放进表格；不要把完全不同的控制流硬塞进一个布尔字段矩阵。

## 3. 可独立运行的最小 demo（Go 1.26）

保存为 `validation_test.go`：

```go
// 可运行测试；Go 1.26.0
package validation

import (
	"context"
	"os"
	"path/filepath"
	"strconv"
	"testing"
)

func parsePort(raw string) (int, error) {
	return strconv.Atoi(raw)
}

func writeFixture(t *testing.T, contents string) string {
	t.Helper()
	path := filepath.Join(t.TempDir(), "port.txt")
	if err := os.WriteFile(path, []byte(contents), 0o600); err != nil {
		t.Fatal(err)
	}
	return path
}

func TestParsePort(t *testing.T) {
	t.Parallel()

	tests := []struct {
		name    string
		raw     string
		want    int
		wantErr bool
	}{
		{name: "http", raw: "80", want: 80},
		{name: "invalid", raw: "x", wantErr: true},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()
			_ = writeFixture(t, tc.raw)

			ctx := t.Context()
			if err := ctx.Err(); err != nil {
				t.Fatalf("unexpected canceled test context: %v", err)
			}

			cleaned := false
			t.Cleanup(func() { cleaned = true })
			t.Cleanup(func() {
				if cleaned {
					t.Error("cleanup order is LIFO")
				}
			})

			got, err := parsePort(tc.raw)
			if (err != nil) != tc.wantErr {
				t.Fatalf("parsePort(%q) error=%v, wantErr=%v", tc.raw, err, tc.wantErr)
			}
			if !tc.wantErr && got != tc.want {
				t.Errorf("parsePort(%q)=%d, want %d", tc.raw, got, tc.want)
			}
		})
	}
}

var _ context.Context // 强调 t.Context 返回标准 context.Context
```

```bash
gofmt -w validation_test.go
go test -v -race .
```

`TempDir` 目录会在测试及其子测试结束后删除；`Cleanup` 按后注册先执行。demo 用第二个 cleanup 验证 LIFO，而不是依赖输出顺序。

## 4. `t.Parallel` 的真实时序

- 调用 `t.Parallel()` 的测试会暂停，等待其顺序父测试函数返回后再与其他并行测试继续；
- 父测试若也 `t.Parallel()`，它可与其他顶层并行测试并发；
- 同一测试进程中的全局变量、环境变量、固定端口、工作目录仍然共享；
- `t.Setenv` 与 `t.Chdir` 影响进程全局状态，不能与并行测试混用；标准库会主动限制部分危险组合；
- `-race` 检查数据竞争，不证明业务时序正确，也不保证 goroutine 已退出。

### 循环变量捕获

Go 1.22 起，`for _, tc := range tests` 用 `:=` 声明时，每轮有独立迭代变量；NGF 的 `go 1.26.0` 已采用该语义。旧代码常见的 `tc := tc` 在当前模块通常不再必需，但若变量在循环外声明后用 `=` 赋值，或闭包捕获共享指针字段，风险仍存在。

## 5. 五个生命周期工具

### `t.Helper`

测试辅助函数第一行调用，失败位置会归因到调用者。NGF `internal/controller/nginx/config/validation/framework_test.go:runValidatorTests` 和 `createGrpcContext` 都这样做。

### `t.TempDir`

每次调用创建独立目录并自动删除。NGF filewatcher 与 gRPC 证书测试用它隔离路径；比手写 `/tmp/test` 更能并行。

### `t.Context`

返回随测试结束而取消的 context。把它传进被测 goroutine/服务，让超时、失败或完成都能传播取消。NGF `command_test.go:createGrpcContext` 以 `t.Context()` 作为 gRPC 上下文父级。

### `t.Cleanup`

即使测试 Fatal/Skip 也会清理，按 LIFO 执行。适合还原全局 hook、关闭 server、取消 context。创建资源的 helper 应就地注册清理，所有权最清楚。

### `t.Run`

把 setup/act/assert 包在案例节点内，确保每例资源和 cleanup 绑定同一生命周期。

## 6. NGF 的两种标准测试形态

`cmd/gateway/validation_test.go:TestEnsureNoPortCollisions` 是最小形态：顶层 `t.Parallel()`，直接断言两个唯一端口成功、重复端口失败。案例少且断言相同，未来若扩展重复位置、多个重复和空输入，可转表格。

更完整的证据是 `internal/controller/nginx/agent/command_test.go:TestSetInitialConfig_Errors`：

```text
tests 表：name + setup(fake, deployment) + errString
  → t.Run(test.name)
  → 子测试 t.Parallel
  → 每例新建 FakeMessenger/client/service/deployment
  → setup 配置 SendReturns / ErrorsReturns / ReturnsOnCall
  → 调 setInitialConfig(t.Context(), ...)
  → 断言错误包含 errString
```

关键不是“用了表”，而是每个并行案例都重新创建可变 fake 和 channel。若把 `FakeMessenger` 放在循环外共享，生成 fake 虽有锁，Returns 配置和调用序列仍会相互污染测试语义。

## 7. 常见失败与迁移边界

- 子测试名重复会自动加序号，定位差；给稳定、描述行为的名字；
- setup 闭包过大时表格变成隐藏控制流，拆成独立测试更清楚；
- 并行测试抢固定端口/全局 logger 会 flaky；用随机监听地址、TempDir 或取消并行；
- `time.Sleep` 等待异步结果会又慢又不稳定；用 channel、Eventually 或带超时轮询；
- helper 忘记 `t.Helper` 会把错误指向辅助函数内部；
- Cleanup 不能掩盖主断言失败，清理错误应独立报告。

## 8. 练习与检查点

1. 为什么父测试中的共享 fake 不适合并行子测试？配置和调用记录跨案例污染，锁只防 race、不提供案例隔离。
2. `t.Context()` 何时取消？测试结束前进入清理阶段时由 testing 管理；被测后台任务应监听它。
3. Cleanup 顺序为何是 LIFO？后创建资源通常依赖先创建资源，应先释放依赖者。
4. 哪些案例不应强行表驱动？setup/断言/失败语义完全不同、共享步骤很少的案例。
