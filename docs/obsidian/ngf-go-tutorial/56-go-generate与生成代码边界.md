# 56 go:generate 与生成代码边界

> [!abstract] 核心结论
> `//go:generate` 是源码中的显式生成命令，不会在 build/test 时自动运行。生成物的 source of truth 是接口/schema + 指令 + 锁定工具版本。修改输入后要重新生成、检查 diff、编译并跑 consumer 测试；带 `Code generated ... DO NOT EDIT.` 的文件不可手改。

## 学习目标与前置

前置：[[24-接口的隐式实现]]、[[25-小接口与依赖注入]]、[[57-表驱动子测试与并行测试]]。完成后应能：

- 写出合法 `//go:generate command args...` 并显式运行；
- 解释工作目录、环境变量、package 选择和 shell 边界；
- 追踪 NGF Counterfeiter 的 interface 输入、directive、tool version 和 fake 输出；
- 修改 interface 后完成“generate → diff → format/compile/test”同步闭环；
- 区分生成代码、手写 adapter 与运行时反射。

## 1. 指令语法与触发

指令必须是 Go 源码中的精确行注释：

```go
//go:generate go run ./cmd/gen
```

`go generate` 只扫描选择到的 package 源文件并执行命令。它不是编译指令，`go build`、`go test`、`go install` 都不会自动触发。

常用范围：

```bash
go generate ./internal/framework/file
go generate ./...
go generate -run counterfeiter ./internal/...
```

`./...` 可能昂贵且修改大量生成物；局部变更先运行最小 package，再由 CI/仓库目标做全量一致性检查。

## 2. 可独立运行 demo：最小 generator

建立目录，放入以下两个输入文件。

**`main.go`（可运行示例的一部分，Go 1.26.0）：**

```go
package main

import "fmt"

//go:generate go run ./cmd/gen

func main() {
	fmt.Println(generatedMessage)
}
```

**`cmd/gen/main.go`（generator）：**

```go
package main

import "os"

func main() {
	const source = "// Code generated by cmd/gen. DO NOT EDIT.\n\npackage main\n\nconst generatedMessage = \"hello generated\"\n"
	if err := os.WriteFile("generated.go", []byte(source), 0o644); err != nil {
		panic(err)
	}
}
```

依次运行：

```bash
go mod init example/generate-demo
go generate .
gofmt -w generated.go
go run .
```

预期输出 `hello generated`。该多文件 demo 已在 `go1.26.0` 的隔离临时目录实际执行；删除 `generated.go` 后直接 `go run .` 会编译失败，证明 generate 不会自动发生。

## 3. 工作目录、参数与环境

命令工作目录是包含 directive 的 package 目录。Go 会设置 `$GOFILE`、`$GOLINE`、`$GOPACKAGE`、`$GOARCH`、`$GOOS` 等；变量展开由 go generate 完成。

命令不是默认交给 shell 解释。管道、重定向、通配等需要显式 `sh -c`，这会降低跨平台性并增加 quoting 风险；更可靠的做法是写 Go generator 程序。

`//go:generate -command name command...` 可在当前文件后续 directives 中定义别名。生成命令有环境权限，不能对不可信仓库盲目执行。

## 4. 工具版本必须锁定

NGF `go.mod` 使用 Go tool dependency：

```go
tool github.com/maxbrunsfeld/counterfeiter/v6
```

模块依赖固定到 `v6.12.2`，源码指令调用 `go tool counterfeiter`，因此开发者和 CI 使用同一模块图工具版本。仅在 README 写“安装最新版”会导致生成 diff 漂移。

升级 generator 是独立变更：先更新 module/tool version，再全量 regenerate，审查行为与机械 diff，并运行相关测试。

## 5. NGF Counterfeiter 输入到输出

以 `internal/framework/file/file.go` 为例：

```go
//go:generate go tool counterfeiter -generate

//counterfeiter:generate . OSFileManager
type OSFileManager interface {
	Create(string) (*os.File, error)
	Chmod(*os.File, os.FileMode) error
	Write(*os.File, []byte) error
	Open(string) (*os.File, error)
	Copy(io.Writer, io.Reader) error
}
```

第一行告诉 go generate 启动工具并扫描当前 package；第二行告诉 Counterfeiter 输入 interface 是当前 package 的 `OSFileManager`。输出是 `internal/framework/file/filefakes/fake_osfile_manager.go`，首行：

```go
// Code generated by counterfeiter. DO NOT EDIT.
```

生成 Fake 提供 Stub、CallCount、ArgsForCall、Returns/ReturnsOnCall 和调用记录锁。测试据此注入 Open/Create/Copy/Chmod 错误，而无需真实文件系统。

## 6. 为什么不可手改生成文件

手改 fake 的问题不是风格，而是 source of truth 分裂：下次 go generate 会覆盖；其他开发者无法从接口和指令重现；多个 fake 可能不同步。

正确修改路径：

1. 改 `OSFileManager` interface 或 directive/tool；
2. 在其 package 运行 go generate；
3. 审查生成文件只反映预期方法变化；
4. 运行 gofmt（generator 通常已格式化，仍可检查）；
5. 编译实现 `StdLibOSFileManager`；
6. 更新 fake 使用方并运行 tests。

若生成器本身输出错误，应修 generator/template 或升级工具，不在输出上打补丁。

## 7. 同步边界不止一个文件

新增 `OSFileManager.Sync` 时至少影响：

- interface 输入；
- `StdLibOSFileManager` 具体实现；
- `FakeOSFileManager` 生成输出；
- 所有构造 literal 中对 Stub/Returns 的使用；
- 调用方错误包装与测试；
- 可能的权限/持久化语义文档。

编译期接口实现会发现缺具体方法；生成 fake 的 consumer 测试发现调用契约变化。只看 generated diff 不足以证明行为正确。

## 8. 仓库中的两类生成

Counterfeiter 生成测试替身；`internal/controller/telemetry/collector.go` 还有 telemetry exporter generator，按 Data/NGFResourceCounts 生成 schema/attributes。不同 generator 的输入与同步清单不同：

| generator | 输入 owner | 输出用途 | 修改后验证 |
|---|---|---|---|
| Counterfeiter | interface/method set | 测试 fake | compile + consumer tests |
| telemetry generator | struct fields/tags/options | telemetry schema/attributes | generated tests/schema compatibility |

不要把所有 `*_generated.go` 当同一机制；先读 directive 和生成文件 header。

## 9. CI 与可重现检查

典型一致性门：在 clean tree 运行生成目标，然后 `git diff --exit-code`。本轮文档修改没有运行仓库全量 go generate，因为它会写分配范围外的生产生成物；证据来自锁定 directive、工具依赖和当前 generated header，相关 package tests用于验证当前同步状态。

这不是声称“全仓生成无 diff”。若交付修改了生成输入，应由授权范围内的工作流实际运行并报告 diff。

## 10. 常见误区与迁移边界

> [!warning] go generate 会执行代码
> 先审查 directive 与依赖来源；不要对来源不明的仓库直接全量执行。

- generate 成功只证明命令退出 0，不证明生成行为正确；
- 删除生成文件后 build 失败是正常的 checked-in codegen 模型；
- 不要依赖开发者全局 PATH 中的“某个版本”工具；
- 生成物很大时仍要审查接口签名、header、package、并发保护等关键变化。

**直接迁移：** 锁工具版本、生成物带 header、clean diff 门。**条件迁移：** checked-in 或构建时生成取决于发布/工具链约束。**不要复制：** 手改 generated fake。

## 11. 练习与检查点

1. 给 demo 新增 `generatedNumber`，只改 generator，重新 generate。检查 generated diff 可复现。
2. 给 `OSFileManager` 设计 `Sync`，列出上述六类同步点，不实际修改源码。
3. 比较 Counterfeiter 和 telemetry generator directive，指出各自输入和失败爆炸半径。
