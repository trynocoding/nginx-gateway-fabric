# 19 函数值、闭包与高阶函数

> [!abstract] 核心结论
> 函数在 Go 中可以赋值、保存、传参和返回。闭包是“函数代码 + 它捕获的外层变量”；高阶函数则接收或返回函数。它们把行为变成依赖，但也把状态、生命周期和并发责任藏进捕获环境，必须主动审计。

## 学习目标与前置

前置：[[18-可变参数]]、理解变量作用域。完成后应能：

- 声明函数类型并区分命名函数、匿名函数、方法值；
- 解释闭包捕获的是变量而非创建瞬间的值；
- 使用 callback、工厂函数与策略函数三种模式；
- 读懂 NGF 的 leader runnable 如何保存 `[]func(context.Context)` 并在框架回调时执行。

## 1. 函数值和函数类型

```go
type Filter func(string) bool

var f Filter = strings.HasPrefix // 签名不匹配，这里只是提醒：参数必须完全一致
```

函数类型由参数和结果决定。命名函数可赋给变量；匿名函数字面量可就地创建；`obj.Method` 是绑定接收者的方法值，`Type.Method` 是还需要显式接收者的方法表达式。

## 2. 可独立运行的最小 demo

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import "fmt"

type Predicate func(int) bool

func keep(values []int, predicate Predicate) []int {
	var result []int
	for _, value := range values {
		if predicate(value) {
			result = append(result, value)
		}
	}
	return result
}

func atLeast(min int) Predicate {
	return func(value int) bool {
		return value >= min
	}
}

func main() {
	predicate := atLeast(3)
	fmt.Println(keep([]int{1, 3, 2, 5}, predicate))
}
```

运行 `go run main.go`，预期输出 `[3 5]`。闭包让返回的函数在 `atLeast` 已结束后仍能读取 `min`。该示例已在 `go1.26.0` 执行验证。

## 3. 捕获语义：最容易误判的地方

闭包捕获变量。外层变量之后改变，闭包观察到的也会改变：

```go
limit := 3
f := func(v int) bool { return v >= limit }
limit = 5 // f 现在使用 5
```

若要冻结值，就在创建闭包时复制到新的局部变量或通过工厂函数参数传入。闭包逃逸时，编译器可能把捕获变量放到堆上；先以清晰的所有权为准，再用 profile 判断性能。

## 4. 常用模式

### 模式一：callback

调用方注入“完成后做什么”，例如 `func(context.Context)`。优点是边界小；风险是 callback 是否阻塞、是否可重入、失败怎么表达必须另行约定。

### 模式二：策略函数

像 demo 的 `Predicate`，数据流程固定、决策行为可替换。比只有一个方法的临时实现类型更轻。

### 模式三：工厂返回闭包

`atLeast(min)` 把配置绑定为函数。[[27-Functional-Options]] 中 `WithX` 返回 `Option` 也是这一模式。

### 模式四：方法值

```go
send := broadcaster.Send
```

`send` 保存接收者；这也延长接收者可达生命周期。若对象含锁、channel 或大状态，不能把方法值误当成无状态函数。

## 5. NGF 生产实例：成为 leader 后调用一组行为

`internal/framework/runnables/runnables.go:CallFunctionsAfterBecameLeader` 保存：

```go
enableFunctions []func(context.Context)
```

其 `Start(ctx)` 依次执行 `f(ctx)`。**这是生产源码缩略节选，不是独立 demo。** 设计分成两阶段：

1. 构造阶段 `NewCallFunctionsAfterBecameLeader` 接收函数值切片并保存；
2. 执行阶段由 controller-runtime 在 leader 条件满足后调用 `Start`；
3. `Start` 把框架提供的 context 传给每个函数。

该类型不负责并行、重试或聚合错误；函数签名也没有 error 结果。因此一个 callback 想报告失败，只能自己记录/处理，不能由 `Start` 返回。这是明确的迁移边界：需要 fail-fast 时应改成 `func(context.Context) error` 并定义停止策略。

`internal/framework/runnables/runnables_test.go` 验证函数会被调用以及 `NeedLeaderElection()` 返回 true。构造函数没有复制传入 slice；调用方若在启动前修改同一底层数组，行为可能随之改变。当前用法应把它视为构造后不再修改的配置。

另一个真实例子在 `internal/controller/handler.go:objectFilter`：struct 字段 `upsert`、`delete` 保存方法值，使不同资源可复用同一分发流程。

## 6. 边界与失败例

> [!warning] callback API 也需要契约
> 至少说明调用次数、顺序、是否并发、是否允许阻塞、panic/error 如何处理、context 由谁取消。

- nil 函数值可以存储，但调用会 panic；构造阶段应拒绝或跳过 nil；
- 在并发 goroutine 中读写同一捕获变量会产生 data race，仍需锁、channel 或原子操作；
- 闭包捕获大对象可能意外延长其生命周期；
- `defer` 中的闭包若读取命名返回值，会在函数返回前看到最终值，详见 [[20-defer与资源清理]]。

## 7. 练习与检查点

1. 把 demo 的 `min` 改为外层变量并在调用前修改。检查闭包使用新值。
2. 将 `keep` 的 predicate 设为 nil。观察 panic，再给 `keep` 增加显式 nil 检查。
3. 设计 `[]func(context.Context) error` 的执行器。检查点：明确遇到第一个错误停止，还是执行全部并 `errors.Join`。
