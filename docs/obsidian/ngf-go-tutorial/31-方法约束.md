# 31 方法约束

> [!abstract]
> 方法约束是泛型函数体的“能力合同”：约束声明了哪些方法，函数体就能对 `T` 静态调用哪些方法。它适合多个具体类型共享同一行为和同一算法的场景；运行时替换实现仍是普通接口的职责。

## 学习目标与前置

- 能声明只含方法的约束，并组合已有接口；
- 理解值方法、指针方法与类型参数方法集的关系；
- 区分“`T` 受接口约束”和“参数本身是接口值”；
- 追踪 NGF `objectStoreMapAdapter[T client.Object]` 的存储边界与失败方式。

前置：[[24-接口的隐式实现]]、[[29-泛型函数与类型推断]]、[[30-any与comparable]]。

## 1. 从接口行为到约束

接口可以用作普通参数类型：

```go
type Named interface {
	Name() string
}

func PrintName(v Named) { fmt.Println(v.Name()) }
```

也可以约束类型形参：

```go
func NameOf[T Named](v T) string { return v.Name() }
```

两者都只允许调用 `Name()`，但数据模型不同：

- `PrintName` 接收一个接口值，适合在同一集合中混装不同实现；
- `NameOf` 每次实例化保留具体 `T`，适合通用算法的输入输出仍需是同一种静态类型。

方法约束也能组合：

```go
type Keyed interface {
	Name() string
	Namespace() string
}
```

编译器会检查每个类型实参的方法集是否覆盖约束，而不是查找 `implements` 声明。

## 2. 指针接收者是最常见的陷阱

```go
type Resource struct{ name string }

func (r *Resource) Name() string { return r.name }
```

这里 `*Resource` 满足 `Named`，`Resource` 不满足。原因是 `Resource` 的方法集不含仅在 `*Resource` 上声明的方法。普通变量有时能自动取地址调用 `r.Name()`，但这项语法便利不会改变接口实现关系。

```go
// NameOf(Resource{})  // 编译失败
NameOf(&Resource{})    // 正确
```

Kubernetes 对象通常通过指针实现 `client.Object`，因此 NGF 的泛型实参也多是 `*gatewayv1.Gateway` 一类指针。

## 3. 可独立运行的最小 demo

**可运行 demo（Go 1.26.0，保存为 `main.go`）：**

```go
package main

import "fmt"

type Keyed interface {
	Key() string
}

type Store[T Keyed] struct {
	items map[string]T
}

func NewStore[T Keyed]() *Store[T] {
	return &Store[T]{items: make(map[string]T)}
}

func (s *Store[T]) Put(v T) {
	s.items[v.Key()] = v
}

func (s *Store[T]) Get(key string) (T, bool) {
	v, ok := s.items[key]
	return v, ok
}

type Pod struct{ name string }

func (p *Pod) Key() string { return p.name }

func main() {
	s := NewStore[*Pod]()
	s.Put(&Pod{name: "nginx-0"})
	p, ok := s.Get("nginx-0")
	fmt.Println(p.name, ok)

	// _ = NewStore[Pod]() // 编译失败：Pod 的方法集没有 Key
}
```

验证：

```bash
gofmt -w main.go
go run main.go
# nginx-0 true
```

## 4. 常用模式

### 4.1 强类型容器

`Store[T Keyed]` 既能调用 `Key()`，又让 `Get` 返回具体 `T`，调用者无需类型断言。

### 4.2 通用批处理

若多种消息都有 `Validate() error`，可写 `ValidateAll[T Validator]([]T) error`，把遍历控制流复用，同时把失败语境留在泛型函数中。

### 4.3 适配外部框架接口

约束可以直接嵌入上游接口：

```go
type ObjectWithKey interface {
	client.Object
	StableKey() string
}
```

这表示 `T` 必须同时满足 controller-runtime 的对象合同和项目的额外合同。不要重复抄写上游接口的方法列表。

### 4.4 泛型内部恢复具体类型

当入口参数已被擦除为 `client.Object`，可用 `obj.(T)` 恢复泛型容器需要的具体类型。断言失败仍是运行时事件，因此必须明确是返回错误还是 panic。

## 5. NGF：`objectStoreMapAdapter`

**NGF 缩写源码（不是独立 demo）：**

```go
type objectStoreMapAdapter[T client.Object] struct {
	objects map[types.NamespacedName]T
}

func (m *objectStoreMapAdapter[T]) upsert(obj client.Object) {
	t, ok := obj.(T)
	if !ok {
		panic(fmt.Errorf("obj type mismatch: got %T, expected %T", obj, t))
	}
	m.objects[client.ObjectKeyFromObject(obj)] = t
}
```

责任边界：

1. `newObjectStoreMapAdapter` 接收已经按具体 Kubernetes 类型建立的 map；
2. `multiObjectStore` 按 GVK 选择正确的 `objectStore`；
3. 适配器在 `upsert` 边界把宽接口 `client.Object` 断言回 `T`；
4. `client.ObjectKeyFromObject` 调用约束保证存在的对象方法，生成 namespaced-name 键；
5. map 内保持 `T`，`get` 再以 `client.Object` 暴露给统一存储接口。

这里“泛型容器 + 非泛型接口”各司其职：泛型保证每张 map 的值类型；`objectStore` 让上层按 GVK 统一调度。若只用 `map[NamespacedName]client.Object`，错误类型可能更晚才暴露；若只用泛型，上层又无法把不同 `T` 的 store 放入同一 registry。

`internal/controller/state/store_test.go:TestConvertingReferenceGrantStore` 通过 `newConvertingReferenceGrantStore` 间接使用 `objectStoreMapAdapter[*gatewayv1.ReferenceGrant]`，覆盖 upsert/get/delete；错误类型的 panic 测试位于同一测试。生产构造路径在 state 跟踪配置中把各类资源 map 包装后注册给 `multiObjectStore`。

## 6. 编译边界、运行边界与误区

| 边界 | 何时检查 | 本例 |
|---|---|---|
| `T` 是否满足 `client.Object` | 编译期 | 禁止 `objectStoreMapAdapter[int]` |
| `obj` 的动态类型是否就是 `T` | 运行时 | `obj.(T)` 可能失败并 panic |
| GVK 是否映射到正确 store | 运行时 | `multiObjectStore.mustFindStoreForObj` 检查 |

误区：

- 方法约束不能消除所有类型断言；若数据先经过宽接口边界，动态检查仍存在。
- 不要为了“更泛型”把具体业务方法都塞入大接口；约束越大，可用类型越少。
- `T client.Object` 不代表 `T` 一定非 nil；指针类型的零值仍可为 nil。
- 约束中不能调用未列出的方法，即使当前所有调用者碰巧都有该方法。

## 7. 选择与迁移边界

- 选泛型：容器和算法需要保留具体对象类型。
- 选接口：上层需要在同一 registry 动态持有不同 store。
- 选类型断言：只在宽接口进入强类型内部的边界做一次，并立即失败。
- 不选反射：存取不依赖字段名或运行时类型构造，接口方法已经足够。

如果错误对象可能来自用户输入或外部插件，panic 不再合适，应把 `upsert` 改为返回 error，并把错误传播到可恢复边界；当前 NGF 将映射错误视为内部注册/接线不变量破坏。参见 [[39-panic与不变量保护]]。

## 8. 练习与答案

1. 为 demo 增加 `Delete(v T)`，用约束方法取键。
2. 把 `Store[T]` 改成普通 `map[string]Keyed`，列出调用 `Get` 后的差异。
3. 给 `Keyed` 增加 `Validate() error` 后，哪些现有类型会在何时失败？

检查点：

```go
func (s *Store[T]) Delete(v T) {
	delete(s.items, v.Key())
}
```

第 2 题会把结果静态类型降为 `Keyed`，需要断言才能访问 `*Pod` 字段。第 3 题会让不具备 `Validate` 的类型在实例化处编译失败；这说明扩张公共约束有迁移成本，通常应新建更窄的约束或把验证策略作为函数传入。
