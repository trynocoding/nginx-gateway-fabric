# 28 构造函数与不变量建立

> [!abstract] 核心结论
> Go 没有特殊 constructor 语法，`NewT` 只是普通函数。它的价值是把默认值、依赖校验、资源分配和跨字段不变量集中到一个入口，使返回值一拿到就可安全使用。若类型零值本来就有效，不必为了形式强加构造函数。

## 学习目标与前置

前置：[[06-struct与复合字面量]]、[[08-指针nil与可选字段]]、[[13-map-comma-ok与缺失值]]、[[25-小接口与依赖注入]]。完成后应能：

- 选择 `NewT() *T`、`NewT() (T, error)` 或可用零值；
- 在构造阶段处理默认值、校验、规范化与内部 map/channel；
- 区分对象不变量、生命周期副作用与调用方责任；
- 解释 NGF `NewDeploymentBroadcaster` 返回前建立了哪些并发不变量。

## 1. 构造函数不是关键字

常见约定：

```go
func NewClient(...) *Client
func NewClient(...) (*Client, error)
func MustNewClient(...) *Client
```

能失败的解析、I/O 或配置校验应返回 error；`MustX` 只适合启动期常量、测试或失败即程序员错误的场景。返回值还是指针取决于 identity、可变性与复制成本，不取决于 `New` 名称。

## 2. 可独立运行 demo：默认值、校验与不变量

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import (
	"errors"
	"fmt"
)

type Registry struct {
	prefix string
	items  map[string]string
}

func NewRegistry(prefix string) (*Registry, error) {
	if prefix == "" {
		prefix = "default"
	}
	if prefix == "-" {
		return nil, errors.New("invalid prefix")
	}
	return &Registry{
		prefix: prefix,
		items:  make(map[string]string),
	}, nil
}

func (r *Registry) Put(key, value string) error {
	if key == "" {
		return errors.New("empty key")
	}
	r.items[r.prefix+"/"+key] = value
	return nil
}

func main() {
	registry, err := NewRegistry("")
	if err != nil {
		panic(err)
	}
	_ = registry.Put("gateway", "ready")
	fmt.Println(registry.items)
	_, err = NewRegistry("-")
	fmt.Println(err)
}
```

预期先输出含 `default/gateway:ready` 的 map，再输出 `invalid prefix`。构造函数保证 `items` 非 nil、prefix 已规范化；`Put` 维护每次操作的不变量。已在 `go1.26.0` 验证成功与失败路径。

## 3. 构造阶段的四项工作

### 默认值与规范化

把空输入转换成明确默认，或 trim/parse 为内部规范形式。默认必须可测试；不要让不同方法各自重复猜默认。

### 必需依赖校验

nil client、非法范围、互斥字段应尽早失败。跨字段校验发生在所有输入/Options 汇总后，见 [[27-Functional-Options]]。

### 分配零值不可用的内部状态

map 写入前必须 make；channel 的容量与方向体现背压；child context/cancel func 必须成对保存；锁通常无需显式初始化。

### 建立生命周期

构造函数若启动 goroutine、注册 callback 或打开文件，返回值就拥有关闭责任。API 必须提供 cancel/Close 或从 parent context 推导退出。

## 4. 何时让零值可用

`bytes.Buffer`、`sync.Mutex` 的零值可用，降低构造负担。自己的类型若能自然给出安全零语义，也应优先考虑：

- 读操作可返回空；
- 首次写可惰性初始化；
- 没有必需外部依赖；
- 不会默默采用危险默认。

相反，含必需 interface、需启动 worker、map 必须写、多个字段必须一致时，构造函数更有价值。即使提供 New，导出 struct 字段或同 package 字面量仍可能绕过它；真正安全要结合私有字段、方法校验和测试。

## 5. NGF 主实例：`NewDeploymentBroadcaster`

`internal/controller/nginx/agent/broadcast/broadcast.go:NewDeploymentBroadcaster` 的生产逻辑可概括为：

```go
broadcasterCtx, broadcasterCancel := context.WithCancel(ctx)
b := &DeploymentBroadcaster{
	listeners:         make(map[string]storedChannels),
	publishCh:         make(chan NginxAgentMessage),
	subCh:             make(chan storedChannels),
	unsubCh:           make(chan string),
	doneCh:            make(chan struct{}),
	broadcasterCtx:    broadcasterCtx,
	broadcasterCancel: broadcasterCancel,
}
go b.subscriber()
go b.publisher()
return b
```

**这是生产源码的缩略节选，不是独立 demo。** 返回时已经成立的不变量：

1. listeners map 可写，且由 `mu` 保护；
2. publish/subscribe/unsubscribe/completion channel 都非 nil，不会因 nil channel 永久禁用 case；
3. broadcaster context 是调用方 context 的 child，并保存 cancel；
4. subscriber 和 publisher 两个 goroutine 已启动，能消费 channel；
5. subscriber 退出时 deferred cancel 会取消所有 listener child contexts，解除 publisher 等待。

构造函数返回 `*DeploymentBroadcaster` 且不返回 error，因为 `make`、WithCancel 与启动 goroutine 没有可报告的普通错误路径；内存不足属于进程级失败。它要求传入非 nil context，`context.WithCancel(nil)` 会 panic，这是调用方前置条件。

## 6. 生命周期与并发迁移边界

该构造函数不仅“填字段”，还启动后台活动。所有阻塞 select 都必须能被 broadcaster/listener context 取消，否则取消订阅或 shutdown 时会泄漏 goroutine。`internal/controller/nginx/agent/broadcast/broadcast_test.go` 覆盖：

- 单/多 listener 订阅与应答；
- 无 listener；
- 取消订阅解除 publisher 等待；
- parent context 取消后忽略消息并退出；
- listener 已收消息时 shutdown 仍能解阻塞。

**可直接迁移：** 返回前初始化全部 channel/map；child context 管理 worker。**有条件迁移：** 构造时启动 goroutine，必须同时设计停止协议和测试。**不要复制：** 只启动 goroutine 却没有 cancellation/Close。

## 7. 补充实例：可失败构造函数

`internal/controller/provisioner/handler.go:newEventHandler` 先用 `metav1.LabelSelectorAsSelector` 解析 selector；失败返回包装 error，成功才构造 handler。它与 broadcaster 展示两类选择：

| 构造内容 | 签名 | 原因 |
|---|---|---|
| 内存状态与 goroutine | `*DeploymentBroadcaster` | 无普通可恢复构造错误 |
| 外部格式解析/校验 | `(*eventHandler, error)` | label selector 可能非法 |

## 8. 构造函数测试矩阵

至少验证：

| 类别 | 检查 |
|---|---|
| 默认值 | 空/省略输入得到文档化默认 |
| 非法输入 | 返回 error 或约定 panic，不返回半有效对象 |
| 内部不变量 | map/channel/context 非 nil，字段关系一致 |
| 行为可用 | 构造后立即调用首个公共方法成功 |
| 生命周期 | cancel/Close 后 goroutine、timer、连接退出 |
| 注入 | fake 依赖被保存且错误能传播 |

只断言私有字段会使测试脆弱；优先通过公共行为证明不变量，再对并发 channel/context 做必要白盒检查。

## 9. 常见误区

> [!warning] `NewT` 不会阻止 `T{}`
> 只有未导出字段、package 边界和方法校验共同作用，才能把外部调用者引向构造入口。同 package 代码仍能绕过。

- 构造函数做网络 I/O 会增加启动延迟与测试复杂度；可拆成 Construct/Start；
- 返回半初始化对象加 error 容易误用，除非明确支持部分结果；
- 默认值散落在构造函数、方法和 YAML 三处会漂移；指定唯一 owner；
- 构造后保存调用方 map/slice 时要决定 clone 或共享，并写进契约。

## 10. 练习与检查点

1. 为 demo 写表驱动测试：空 prefix、显式 prefix、非法 prefix、空 key。检查构造不变量和操作不变量分开。
2. 设计 broadcaster 的 `Close()`。检查多次调用是否幂等、如何等待两个 goroutine 退出，以及与 parent cancel 的关系。
3. 给 `newEventHandler` 传非法 selector。检查它在启动任何 goroutine/产生副作用前返回 error。
