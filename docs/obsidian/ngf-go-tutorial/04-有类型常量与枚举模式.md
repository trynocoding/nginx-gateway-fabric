# 04 有类型常量与枚举模式

> [!abstract]
> Go 没有封闭枚举。常用做法是“定义命名类型 + 声明一组有类型常量 + 在边界显式校验”。类型系统能防止不同概念意外混用，但不能阻止构造集合外的新值。

## 学习目标与前置

学完应能：

- 区分无类型常量、有类型常量和变量；
- 用 string 或整数底层类型建立枚举式 API；
- 知道 `iota` 适合什么、不适合什么；
- 解释为何 NGF 同时需要 Go 常量、Kubebuilder Enum 标记和运行时 `switch`。

前置：[[03-变量声明与零值]]。命名类型与显式转换在 [[05-类型定义别名与显式转换]] 展开。

## 1. 常量的基本语法

```go
const defaultPort = 80          // 无类型整数常量
const securePort int32 = 443    // 有类型常量

type Mode string

const (
	ModeSafe Mode = "safe"
	ModeFast Mode = "fast"
)
```

常量必须能在编译期求值，不能写 `const now = time.Now()`。无类型常量在赋给具体上下文时才确定默认/目标类型；有类型常量从声明起就携带命名类型。

### 枚举模式并不封闭

下面仍能编译：

```go
var m Mode = "experimental"
```

所以常量集合提供“推荐的、可发现的名字”，不是完整合法性证明。外部输入必须校验。

### `iota` 是常量生成器

```go
type State uint8

const (
	StatePending State = iota
	StateRunning
	StateDone
)
```

它适合进程内、不需要稳定文本协议的连续整数。若值写入 JSON、磁盘、网络或 Kubernetes API，显式字符串通常更容易阅读，也避免插入常量改变后续数值。

## 2. 可独立运行的最小 demo（Go 1.26）

```go
// 可运行示例；Go 1.26.0
package main

import (
	"fmt"
	"slices"
)

type LogLevel string

const (
	LogInfo  LogLevel = "info"
	LogDebug LogLevel = "debug"
	LogError LogLevel = "error"
)

var validLevels = []LogLevel{LogInfo, LogDebug, LogError}

func ParseLogLevel(raw string) (LogLevel, error) {
	level := LogLevel(raw)
	if !slices.Contains(validLevels, level) {
		return "", fmt.Errorf("unsupported log level %q", raw)
	}
	return level, nil
}

func main() {
	for _, raw := range []string{"debug", "warn"} {
		level, err := ParseLogLevel(raw)
		fmt.Printf("raw=%q level=%q err=%v\n", raw, level, err)
	}
}
```

运行：

```bash
gofmt -w main.go
go run main.go
# raw="debug" level="debug" err=<nil>
# raw="warn" level="" err=unsupported log level "warn"
```

失败检查：删除 `ParseLogLevel` 的校验，`LogLevel("warn")` 完全合法。这正说明命名类型不能替代输入验证。

## 3. 四种常用模式

### 3.1 string 枚举用于人类可见配置

配置文件和 API 中直接看到 `debug`，兼容演进清晰。代价是占用略多、必须校验拼写。

### 3.2 `iota` 状态机用于进程内状态

适合状态只在内存使用且不依赖具体数值。通常保留零值给 `Unknown`，让未初始化状态可辨识。

### 3.3 `switch` 做穷尽式行为分派

```go
switch level {
case LogInfo:
	// ...
case LogDebug:
	// ...
default:
	return fmt.Errorf("unsupported: %q", level)
}
```

Go 编译器不会检查是否覆盖全部常量，所以 `default` 与测试仍重要。

### 3.4 枚举值集中定义，边界分层校验

API schema 尽早拒绝非法远程输入；进程内函数再次校验关键不变量；下游只消费验证后的类型。不要在多个目录复制不同 allowlist。

## 4. NGF：`ControllerLogLevel` 的三道边界

焦点类型：`ngf:apis/v1alpha1/nginxgateway_types.go:ControllerLogLevel`。

**NGF 缩写源码（不是独立 demo）：**

```go
// +kubebuilder:validation:Enum=info;debug;error
type ControllerLogLevel string

const (
	ControllerLogLevelInfo  ControllerLogLevel = "info"
	ControllerLogLevelDebug ControllerLogLevel = "debug"
	ControllerLogLevelError ControllerLogLevel = "error"
)
```

三层各自负责：

| 层 | 机制 | 保证与限制 |
|---|---|---|
| Go 源码 | 命名类型 + 有类型常量 | 防止与其他 string 类型随意混用；不封闭集合 |
| Kubernetes schema | Kubebuilder `Enum` 标记生成 CRD 校验 | API server 可拒绝非法值；仅覆盖经过 API 的输入 |
| 控制器运行时 | `validateLogLevel` 的 `switch` | 保护内部调用、测试对象或绕过 schema 的路径 |

数据关系：

```text
NginxGateway.spec.logging.level 字符串
  → JSON 解码为 *ControllerLogLevel
  → updateControlPlane 合并默认 info
  → validateLogLevel switch
  → 转成 string 交给 logLevelSetter.SetLevel
```

`validateLogLevel` 只接受三个常量，默认分支返回 Kubernetes `field.NotSupported`。`internal/controller/config_updater_test.go:TestValidateLogLevel` 覆盖三种合法值与 `invalid/high/warn`；`TestUpdateControlPlane` 还证明非法值不会调用 setter。

### 为什么不是 `iota`

该值出现在 CRD YAML/JSON 中，显式字符串是稳定的外部协议。若用 0/1/2，用户可读性差，而且调整顺序会制造兼容风险。

### 默认值不是零值

`ControllerLogLevel` 的零值是 `""`，不是 `info`。`updateControlPlane` 先构造 `Level: &ControllerLogLevelInfo`，再覆盖用户配置。这提醒我们：语言零值与业务默认值是两层概念。

## 5. 边界、误区与迁移

- **直接复用**：对外部文本协议使用命名 string 常量，并在入口校验。
- **条件复用**：重复运行时校验，适合存在多输入路径或不完全信任生成 schema 的系统。
- **不可照搬**：仅声明常量便声称值域封闭；调用者仍可转换任意 string。
- 新增 `warn` 时至少同步：Go 常量、Kubebuilder Enum、`validateLogLevel` allowlist、setter 支持与测试。
- 不要依赖 map 遍历顺序生成“支持值”错误；若错误文本需要稳定，使用有序 slice。

## 6. 练习与答案

1. 为什么 `const X = []int{1}` 不合法？slice 需要运行时对象，不是常量表达式。
2. `Mode("other")` 为什么能编译？显式转换允许构造底层类型可表示的值，命名类型不是封闭 enum。
3. 新增日志级别最容易漏哪一层？schema、运行时 switch、下游 setter、测试都可能漏，应按同步清单修改。
4. 何时给 `iota` 的第一个值命名为 `Unknown`？当零值可能自然出现且需要与有效状态区分时。
