# 32 类型集合、`~` 与联合约束

> [!abstract]
> 约束接口不仅能列方法，还能描述允许的类型集合。`A | B` 是并集；`~T` 接受底层类型为 `T` 的所有命名类型。它们适合“操作由底层表示决定”的封闭算法，而不是运行时的类型分派。

## 学习目标与前置

- 从类型集合角度解释 `~string`、`~int | ~int64`；
- 区分定义类型、别名和底层类型；
- 知道约束接口只能用于类型形参，不能拿来声明普通运行时变量；
- 理解 NGF 为什么用 `T ~string` 合并 Gateway API 的两种命名枚举。

前置：[[05-类型定义别名与显式转换]]、[[29-泛型函数与类型推断]]、[[31-方法约束]]。

## 1. 三种容易混淆的类型关系

```go
type HeaderKind string     // 新的定义类型，底层类型是 string
type QueryKind string      // 另一个定义类型
type Text = string         // 别名，与 string 完全相同
```

`HeaderKind` 与 `QueryKind` 不能无条件互相赋值，需要显式转换；但它们都具有 `string` 的底层表示和相应运算。

若约束写成：

```go
type ExactString interface {
	string
}
```

类型集合只包含精确的 `string`，不包含 `HeaderKind`。加上波浪号：

```go
type StringLike interface {
	~string
}
```

集合扩展为所有底层类型是 `string` 的类型，因而三者都可作为类型实参。

## 2. 联合约束是类型集合的并集

```go
type Integer interface {
	~int | ~int32 | ~int64
}
```

`|` 不是值层面的逻辑或，而是类型项的联合。函数体只能使用集合中所有类型都支持、且规范允许的共同操作：

```go
func Add[T Integer](a, b T) T { return a + b }
```

类型项和方法也能组合：

```go
type StringEnum interface {
	~string
	String() string
}
```

这要求类型既以 string 为底层类型，又具有 `String()` 方法。

> [!warning]
> 含 `~` 或联合类型项的接口是约束接口，不能写 `var x StringLike` 作为普通值类型。它描述可实例化的类型集合，不是运行时动态值的容器。

## 3. 可独立运行的最小 demo

**可运行 demo（Go 1.26.0，保存为 `main.go`）：**

```go
package main

import "fmt"

type HeaderKind string
type QueryKind string
type Port int32

type StringLike interface {
	~string
}

type SmallNumber interface {
	~int | ~int32
}

func Normalize[T StringLike](v T) string {
	return "kind=" + string(v)
}

func Twice[T SmallNumber](v T) T {
	return v + v
}

func main() {
	fmt.Println(Normalize(HeaderKind("Exact")))
	fmt.Println(Normalize(QueryKind("RegularExpression")))
	fmt.Println(Twice(Port(443)))

	// Normalize([]byte("x")) // 编译失败：[]byte 的底层类型不是 string
}
```

验证：

```bash
gofmt -w main.go
go run main.go
# kind=Exact
# kind=RegularExpression
# 886
```

## 4. 常用模式

### 4.1 合并多个命名枚举

多个 API 定义 `type X string`，且转换规则只依赖字符串常量时，`T ~string` 可以复用一份算法，同时保留调用处的具体类型检查。

### 4.2 数值算法

`~int | ~int32 | ~int64` 让领域类型如 `type Port int32` 保留名称，又能复用加减比较算法。若需要所有整数和浮点，不应手写不完整清单，可使用合适的标准/项目约束。

### 4.3 保留结果命名类型

`func Twice[T SmallNumber](v T) T` 返回原来的命名类型；若签名改成接收 `int64`，调用者会失去 `Port` 类型或被迫反复转换。

### 4.4 封闭类型族

联合约束适合项目明确支持的有限表示，例如 `~string | ~[]byte` 的编码输入。但若新类型需要完全不同的行为，应该用接口方法或策略函数，而不是继续扩联合。

## 5. NGF：两种 Gateway API 枚举共用转换

`HTTPHeaderMatch.Type` 和 `HTTPQueryParamMatch.Type` 的静态类型不同：分别是 `v1.HeaderMatchType` 与 `v1.QueryParamMatchType`，但两者底层都是 string，并共享 `Exact`、`RegularExpression` 语义。

**NGF 缩写源码（不是独立 demo）：**

```go
func convertMatchType[T ~string](matchType *T) MatchType {
	switch *matchType {
	case T(v1.HeaderMatchExact), T(v1.QueryParamMatchExact):
		return MatchTypeExact
	case T(v1.HeaderMatchRegularExpression), T(v1.QueryParamMatchRegularExpression):
		return MatchTypeRegularExpression
	default:
		panic(fmt.Sprintf("unsupported match type: %v", *matchType))
	}
}
```

调用关系：

1. `convertMatch` 遍历 `HTTPRouteMatch.Headers` 和 `QueryParams`；
2. header 分支传 `*v1.HeaderMatchType`，推断 `T` 为 `v1.HeaderMatchType`；
3. query 分支传 `*v1.QueryParamMatchType`，推断另一个实例化；
4. 两者都转换成 NGF dataplane 的 `MatchType`；
5. 后续配置生成只面对统一内部模型。

函数内的 `T(...)` 是显式类型转换，让来自两个枚举族的常量进入当前 `T` 后再参与 switch 比较。这里选择 `~string` 而不是 `string`，正是为了接受命名枚举；选择泛型而不是 `any`，是为了拒绝非 string 底层类型且避免运行时断言。

`internal/controller/state/dataplane/convert_test.go:TestConvertMatchType` 对 header/query 两个实例化分别验证 exact、regex 和 unsupported panic；`TestConvertMatch` 则从完整 route match 验证结果进入 dataplane 模型。

## 6. 运行边界与失败模式

这段实现还有两个前置不变量：

- `matchType` 必须非 nil，否则解引用会 panic；调用侧依赖上游默认化/图构建保证；
- 字符串值必须在受支持集合中，否则 default panic。

`T ~string` 只保证表示是 string，不保证值一定合法。类型约束解决“可进行哪些操作”，值验证仍需 switch、校验函数或 schema。

失败例：

```go
type Bytes []byte
// convertMatchType((*Bytes)(nil)) // 编译失败：Bytes 不满足 ~string

type Other string
v := Other("unknown")
// 若函数可见且调用，会通过编译但在运行时进入 default panic。
```

> [!note]
> 如果 `Other("unknown")` 来源于用户输入，返回 error 比 panic 更合适。NGF 此处的值已穿过 Gateway API 校验，并把未知值视为内部不变量破坏；边界选择详见 [[39-panic与不变量保护]]。

## 7. 泛型、接口与 type switch 的选择

| 问题 | 工具 |
|---|---|
| 同一运算适用于同底层类型 | `~T` 约束 |
| 只支持有限表示族 | 联合约束 |
| 不同类型提供同一行为 | 方法接口 |
| 运行时值属于少数封闭类型 | type switch |
| 读取运行时字段/标签 | 反射，且应隔离在边界 |

迁移边界：如果新增 match 类型只是另一个 `type X string` 且语义完全共享，可继续复用；如果新类型的合法常量或默认规则不同，应拆分转换器或传入策略，避免一个宽约束掩盖领域差异。

## 8. 练习与答案

1. 去掉 demo 中 `StringLike` 的 `~`，观察哪两个调用失败。
2. 写 `Max[T ~int | ~int32](a, b T) T`。
3. 为什么不能用 `var x StringLike = "a"`？

检查点：

```go
func Max[T ~int | ~int32](a, b T) T {
	if a > b {
		return a
	}
	return b
}
```

第 1 题中两个命名 string 类型都不再满足精确 `string` 类型项。第 3 题中 `StringLike` 是只允许用于约束的类型集合接口，不是基本接口；运行时多态应使用只含方法的普通接口。
