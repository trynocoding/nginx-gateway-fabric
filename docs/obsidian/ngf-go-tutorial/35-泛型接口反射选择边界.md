# 35 泛型、接口、type switch 与反射的选择边界

> [!abstract]
> 四种工具回答不同问题：泛型复用编译期已知类型的同形算法；接口抽象行为；type switch 对运行时封闭类型集分支；反射读取或构造运行时类型元数据。先判断“类型何时已知”和“变化的是数据还是行为”，再选工具。

## 学习目标与前置

- 用决策问题而非个人偏好选择机制；
- 理解泛型函数中仍可能存在接口断言边界；
- 知道反射 panic 的典型条件和隔离方式；
- 对比 NGF `MustCastObject` 与 `Reconciler.mustCreateNewObject`。

前置：[[24-接口的隐式实现]]、[[26-类型断言与type-switch]]、[[29-泛型函数与类型推断]]、[[54-reflect与运行时类型注册]]。

## 1. 最小心智模型

| 工具 | 类型何时确定 | 主要能力 | 典型失败 |
|---|---|---|---|
| 泛型 | 编译期实例化 | 保留 T、复用算法 | 约束不满足，编译失败 |
| 接口 | 运行时动态类型 | 按方法合同替换实现 | 断言失败或 nil 语义 |
| type switch | 运行时 | 对已知有限类型分类 | 漏掉 default/新增类型 |
| reflect | 运行时元数据 | 动态构造、字段/标签访问 | 多种运行时 panic |

泛型与接口不是竞争关系：泛型内部可以接受接口约束，系统外层也可用接口持有不同泛型实例的适配器。

## 2. 可独立运行的最小 demo

**可运行 demo（Go 1.26.0，保存为 `main.go`）：**

```go
package main

import (
	"fmt"
	"reflect"
)

type Named interface {
	Name() string
}

type Pod struct{ ID string }

func (p *Pod) Name() string { return p.ID }

func NameOf[T Named](v T) string { return v.Name() }

func Describe(v any) string {
	switch x := v.(type) {
	case string:
		return "string:" + x
	case int:
		return fmt.Sprintf("int:%d", x)
	default:
		return fmt.Sprintf("other:%T", v)
	}
}

func NewLike[T any](prototype *T) *T {
	t := reflect.TypeOf(prototype).Elem()
	return reflect.New(t).Interface().(*T)
}

func main() {
	p := &Pod{ID: "nginx-0"}
	fmt.Println(NameOf(p))
	fmt.Println(Describe(42))
	fmt.Printf("%T\n", NewLike(p))
}
```

验证：

```bash
gofmt -w main.go
go run main.go
# nginx-0
# int:42
# *main.Pod
```

demo 的 `NewLike` 只是教学展示；已知 `T` 时直接 `new(T)` 更简单。反射只有在真实 API 把 prototype 作为运行时接口值时才有价值。

## 3. 四种常用模式

### 3.1 同形集合算法：泛型

`Map[T]`、`Page[T]`、typed pointer helper 都在编译期知道具体类型，优先泛型。

### 3.2 可替换依赖：小接口

文件管理器、getter、logger 的调用者只关心行为，且测试/生产实现不同，优先消费方定义的小接口。

### 3.3 事件联合：type switch

事件通道传 `any`，运行时可能是 Upsert/Delete 等封闭集合，type switch 让每个 case 恢复具体类型；default 应暴露非法接线。

### 3.4 框架注册与动态构造：反射

controller 注册时只有一个 `client.Object` prototype，需要按其运行时具体类型创建新零值对象；此时 `reflect.TypeOf`/`New` 是合适边界。

## 4. NGF 路径一：泛型恢复具体对象

**NGF 缩写源码（不是独立 demo）：**

```go
func MustCastObject[T client.Object](object client.Object) T {
	if obj, ok := object.(T); ok {
		return obj
	}
	panic(fmt.Errorf("unexpected object type %T", object))
}
```

例如 `status_setters.go:newAuthenticationFilterStatusSetter` 接收统一的 `client.Object`，随后调用 `MustCastObject[*ngfAPI.AuthenticationFilter]`，恢复具体类型并写其 Status。

泛型提供“返回值就是目标 T”；接口提供 controller-runtime 的统一对象边界；类型断言负责从宽边界回到领域类型。错误类型意味着 setter 与注册对象接错，因此当前实现 fail-fast。

`internal/framework/helpers/helpers_test.go:TestMustCastObject` 同时覆盖正确 Gateway 断言和错误 BackendTLSPolicy 目标的 panic。

## 5. NGF 路径二：反射创建同类型新对象

`internal/framework/controller/reconciler.go:Reconciler.mustCreateNewObject` 的输入是配置中的 `ngftypes.ObjectType` 接口值：

```go
t := reflect.TypeOf(objectType).Elem()
obj, ok := reflect.New(t).Interface().(client.Object)
if !ok {
	panic("failed to create a new object")
}
```

构造/执行关系：`register` 保存 object prototype → controller-runtime 调 `Reconcile` → `mustCreateNewObject` 按 prototype 动态类型创建零值指针 → Getter 把 Kubernetes API 数据填入对象 → Reconciler 发送 UpsertEvent。

特殊边界：

- `OnlyMetadata` 时不走反射，构造 `PartialObjectMetadata` 并保留 GVK；
- `reflect.TypeOf(objectType).Elem()` 要求 prototype 是非 nil 指针；
- `unstructured.Unstructured` 的 GVK 存在运行时字段中，`reflect.New` 后必须显式恢复；
- `client.Object` 断言失败视为注册不变量破坏。

注释记录了选择反射而非 `DeepCopyObject` 的原因：基准确认后者稍慢。这是源码中的明确说明，不应外推为所有项目的通用性能结论。`internal/framework/controller/reconciler_test.go` 从 `Reconcile` 验证创建出的对象类型、upsert/delete、过滤与取消路径；当前没有独立直测私有反射函数的失败分支。

## 6. 决策顺序

1. 类型在编译期已知、算法同形？用泛型。
2. 调用者只关心一组方法、实现需替换？用接口。
3. 值已进入接口，候选类型少且封闭？用 type switch。
4. 只有运行时元数据能完成注册、构造或编码？把反射隔离在小函数。
5. 若以上都不是，具体代码往往最好。

## 7. 误区与迁移边界

- 用 `any` + reflect 写本可用泛型完成的集合算法，会把编译错误推迟到运行时。
- 用巨型接口替代泛型容器，会丢失具体返回类型并扩大 mock 成本。
- type switch 每增类型都要同步 case 和测试；开放插件集应使用接口/registry。
- 反射前要验证 Kind、nil、可设置性和断言结果；不要让 `reflect.Value` 穿透领域层。
- `Must...` API 只适用于调用者错误/内部不变量。外部数据类型不匹配应返回 error。

迁移时以边界为单位：若 Reconciler 改为构造函数注册，可用 `func() client.Object` 消除反射，但需同步 register 配置和所有构造者；不要只替换函数内部。

## 8. 练习与答案

1. 把 demo `NewLike` 改成 `func New[T any]() *T`，说明为什么不再需要 reflect。
2. 为 `Describe` 新增 bool 后，哪些测试应添加？
3. 设计 `ObjectFactory func() client.Object` 替代 NGF prototype，列出收益和代价。

检查点：

```go
func New[T any]() *T {
	return new(T)
}
```

第 3 题收益是动态构造变为显式函数且容易测试；代价是每个注册点要提供 factory，GVK 初始化仍需负责，并形成 API 迁移。是否值得应以注册规模、性能基准和错误历史决定。
