# 21 方法与接收者

> [!abstract] 核心结论
> 方法是带接收者的函数。接收者把行为放到某个已定义类型的方法集中，但不会自动提供封装、继承或并发安全。先问“行为属于谁、是否修改它、能否复制它”，再选值或指针接收者。

## 学习目标与前置

前置：[[05-类型定义别名与显式转换]]、[[08-指针nil与可选字段]]、[[19-函数值闭包与高阶函数]]。完成后应能：

- 声明并调用方法，区分接收者、普通参数与返回值；
- 使用方法值和方法表达式；
- 解释方法为何只能定义在当前 package 的非指针定义类型上；
- 从 NGF 的 `Reconciler` 读出接收者持有的依赖与一次调用的副作用。

值/指针接收者对接口方法集的精确影响留到 [[22-值与指针接收者方法集]]。

## 1. 基础语法

```go
type Counter struct {
	value int
}

func (c *Counter) Add(delta int) int {
	c.value += delta
	return c.value
}
```

`(c *Counter)` 是接收者列表。`c` 只是局部变量名，通常取类型首字母，不用 `this/self`。方法名和普通函数一样受导出规则约束。

不能给另一个 package 定义的类型直接加方法，也不能以 `*Counter` 作为新定义类型的底层再为“指针定义类型”加方法。需要扩展外部类型时，使用包装 struct、嵌入或普通函数。

## 2. 可独立运行的最小 demo

**可运行示例（Go 1.26.0，标准库）：**

```go
package main

import "fmt"

type Counter struct {
	value int
}

func (c *Counter) Add(delta int) int {
	c.value += delta
	return c.value
}

func (c Counter) Value() int {
	return c.value
}

func main() {
	counter := Counter{}
	fmt.Println(counter.Add(2))

	read := counter.Value       // 方法值：已绑定 counter 的副本
	add := (*Counter).Add        // 方法表达式：接收者仍是第一个参数
	fmt.Println(add(&counter, 3))
	fmt.Println(read(), counter.Value())
}
```

运行 `go run main.go`，预期输出 `2`、`5`、`2 5`。`read` 创建时复制了值接收者，因此保存的是当时值；`add` 则要求显式传 `*Counter`。已在 `go1.26.0` 验证。

## 3. 三种常用组织模式

### 模式一：实体方法

方法读写接收者状态，如 `counter.Add`。不变量应由构造函数和方法共同维护，不能假设“有方法就不会被零值或字面量绕过”。

### 模式二：无状态行为挂到小值类型

枚举或轻量配置可用值接收者，如 `String()`、`Validate()`。调用像领域语言，且复制成本可控。

### 模式三：适配框架接口

框架约定某个方法签名，项目类型实现它后交给框架。例如 controller-runtime 调用 `Reconcile(ctx, req)`；框架决定调用时机，接收者保存项目依赖。

### 方法值作为 callback

`handler.enable` 可作为 `func(context.Context)` 传递。它隐式绑定 handler；要审计这个绑定是否延长生命周期、是否允许并发调用。

## 4. NGF 实例：`(*Reconciler).Reconcile`

`internal/framework/controller/reconciler.go:Reconciler` 只有一个私有 `cfg ReconcilerConfig` 字段。构造后，controller-runtime 通过：

```go
func (r *Reconciler) Reconcile(
	ctx context.Context,
	req reconcile.Request,
) (reconcile.Result, error)
```

调用它。**这是签名化简后的生产源码节选，不是独立 demo。** 一次调用的关键数据关系：

1. 接收者 `r` 提供固定依赖：Getter、对象类型、输出 channel、可选 filter；
2. 参数 `ctx/req` 表示本次 reconcile 的生命周期和资源 key；
3. 方法创建目标对象并调用 Getter；
4. NotFound 被转换为 `DeleteEvent`，存在则转换为 `UpsertEvent`；
5. 发送前同时监听 `ctx.Done()`，避免取消后永久阻塞；
6. 返回 `reconcile.Result` 与 error 给框架。

它使用指针接收者并不只是因为当前方法会直接修改 struct；更重要的是构造、接口断言与框架注册都统一使用 `*Reconciler`，避免复制依赖容器，并为以后增加状态留下稳定身份。`internal/framework/controller/reconciler_test.go` 验证过滤、Get 错误、NotFound、upsert/delete 与 context 取消分支。

## 5. nil 接收者与并发边界

Go 允许对类型为 `*T` 的 nil 指针发起方法调用；是否 panic 取决于方法体是否解引用。不要把这理解为“nil 安全”。若 nil 有业务意义，方法开头应显式处理并写测试；否则让构造函数和调用点保证非 nil。

> [!warning] 方法不等于同步
> 多个 goroutine 调用同一接收者时，字段读写仍需锁、channel ownership 或不可变设计。接收者语法不会消除 data race。

- 复制含 `sync.Mutex` 的值会复制锁状态，通常必须用指针接收者且禁止复制；
- 返回内部 slice/map 会泄漏可变状态，方法边界不自动防御性复制；
- receiver 名称在同一类型的方法之间应一致，降低阅读成本。

## 6. 练习与检查点

1. 把 demo 的 `Value` 改成指针接收者，再观察方法值 `read`。检查：它绑定同一个 counter，最终读到 5。
2. 写 `func (c *Counter) Value() int` 并令 `var c *Counter` 后调用。检查：若访问 `c.value` 会 panic；加 nil 分支才安全。
3. 沿 `Register → cfg.newReconciler → Reconcile` 画三步图。检查：构造发生一次，Reconcile 可被框架调用多次。
