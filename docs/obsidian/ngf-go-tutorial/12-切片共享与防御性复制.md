# 12 切片共享底层数组与防御性复制

> [!abstract]
> 赋值、传参和子切片通常只复制 slice header，元素仍共享。跨越所有权边界时，用 `copy`、`append([]T(nil), src...)` 或 `slices.Clone` 分配新数组；但对含指针/嵌套容器的元素，这仍只是逐元素浅复制。

## 学习目标与前置

- 预测两个 slice 是否可能共享元素；
- 使用三种复制写法并区分 nil/空语义；
- 识别“复制 slice”与“递归深拷贝”的差别；
- 解释 NGF 响应 header 生成为什么复制 Remove。

前置：[[10-数组与切片的类型差异]]、[[11-append容量与预分配]]。

## 1. 共享从哪里产生

```go
source := []string{"A", "B", "C"}
alias := source
window := source[1:]
```

`alias[0] = "X"` 会修改 source；`window[0] = "Y"` 会修改 source[1]。把 slice 传给函数也复制 header，不自动复制元素。

append 更微妙：容量足够时可能继续共享，容量不足时可能分配新数组。不要通过“我刚 append 过”推断隔离。

### 三种复制方法

```go
dst1 := make([]string, len(source))
copy(dst1, source)

dst2 := append([]string(nil), source...)
dst3 := slices.Clone(source)
```

`copy` 返回实际复制元素数；目标长度不足会截断。`slices.Clone(nil)` 保留 nil，而 `make([]T, len(nil))` 得到非 nil 空 slice，这在 JSON/API 语义上可能重要。

## 2. 可独立运行的最小 demo（Go 1.26）

```go
// 可运行示例；Go 1.26.0
package main

import "fmt"

func clone[T any](src []T) []T {
	dst := make([]T, len(src))
	copy(dst, src)
	return dst
}

func main() {
	source := []string{"X-A", "X-B"}
	alias := source
	isolated := clone(source)

	alias[0] = "changed-through-alias"
	isolated[1] = "changed-in-copy"

	fmt.Println("source", source)
	fmt.Println("copy", isolated)
}
```

```bash
gofmt -w main.go
go run main.go
# source [changed-through-alias X-B]
# copy [X-A changed-in-copy]
```

## 3. 常用所有权模式

### 3.1 构造时复制输入

对象要长期保存调用者传入的 slice 时先 clone，防止调用者稍后修改内部状态。

### 3.2 返回时复制内部状态

getter 返回 clone，避免调用者穿透封装。高频热路径需权衡成本，可改成迭代器/只读回调，但 Go 类型系统没有真正只读 slice。

### 3.3 截断容量限制 append 影响

三索引切片 `view := src[:n:n]` 令 len=cap=n，后续 append 必须分配，避免覆盖 src 尾部；已有元素仍共享，修改 `view[i]` 仍影响 src。

### 3.4 对嵌套元素做真正深复制

`[]*Config` clone 后指针仍指同一 Config；`[][]byte` 的内层 slice 仍共享。需要逐元素复制，或调用领域类型 `DeepCopy`。

## 4. NGF：`generateResponseHeaders` 的防御性复制

焦点：`ngf:internal/controller/nginx/config/servers.go:generateResponseHeaders`。

```go
// NGF 原样短源码，不是独立 demo
responseRemoveHeaders := make([]string, len(headerFilter.Remove))
copy(responseRemoveHeaders, headerFilter.Remove)

return http.ResponseHeaders{
	Add:    createHeaders(headerFilter.Add),
	Set:    createHeaders(headerFilter.Set),
	Remove: responseRemoveHeaders,
}
```

`headerFilter.Remove` 属于 dataplane 输入模型；返回的 `http.ResponseHeaders.Remove` 属于 NGINX 配置模型。如果直接赋值，两个层的 slice 会共享数组，后续排序、规范化或模板准备中的修改可能反向污染源模型。

数据关系：

```text
dataplane.HTTPFilters.ResponseHeaderModifiers.Remove
  → 新建等长 []string
  → copy 字符串元素
  → http.ResponseHeaders.Remove
  → 配置生成消费
```

这里注释称 deep copy。对 `[]string` 而言，复制每个 string 值已经足以隔离可观察修改：string 不可变，新的 slice 数组不共享元素槽位。若元素改成 `[]*Header`，相同两行就不再是递归深拷贝。

Add/Set 没直接复用输入 slice，而是 `createHeaders` 逐项构造新的 `[]http.Header`，同样完成模型转换与所有权隔离。

`internal/controller/nginx/config/servers_test.go:TestGenerateResponseHeaders` 验证生成结果；维护时还应保留或补充“修改返回 Remove 不影响输入”的显式别名测试，以固定防御性复制目的。

## 5. 为什么边界处复制值得

Graph/dataplane/NGINX 配置是不同状态层。允许跨层共享可变 slice 会让“哪个层拥有真相”变模糊，并可能产生难复现的顺序依赖或并发问题。复制成本是 O(n)，header remove 列表通常有限；清晰所有权比省一次小复制更重要。性能判断仍应以实际 profile 为准。

## 6. 边界与误区

- `dst := src` 不是元素复制；
- `copy(nil, src)` 复制 0 个，因为目标长度 0；必须先分配长度；
- `append(src[:0], values...)` 会复用并覆盖 src，不是防御性复制；
- clone 只逐元素复制，嵌套引用仍共享；
- nil 与空非 nil slice 在 `len/range` 等价，但序列化/patch 语义可能不同；
- 防御性复制不能替代锁：若另一 goroutine 正在并发修改 source，复制本身仍可能 race。

## 7. 迁移决策

- **直接复用**：跨状态层/封装边界复制可变 slice。
- **条件复用**：`slices.Clone`，先确认是否需要保留 nil。
- **不可照搬**：把逐元素 copy 一律叫深拷贝；要检查元素图。
- 修改元素类型时重新审计所有 clone/copy 点，是防止浅复制回归的关键检查。

## 8. 练习与答案

1. `dst := make([]int, 0, len(src)); copy(dst, src)` 复制几个？0，因为 copy 看 len，不看 cap。
2. 三索引切片能阻止对现有元素的修改吗？不能，只迫使 append 扩容。
3. `[]string` 的逐元素复制为何足够隔离？string 值不可变，新 slice 的元素槽位独立。
4. `[][]byte` 怎么复制？先复制外层，再为每个内层分配并 copy/Clone。
